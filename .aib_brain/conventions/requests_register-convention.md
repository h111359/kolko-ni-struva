## Purpose
Define a precise, deterministic convention for the **requests register** that lists all user-created AIB requests, their lifecycle state, and the workspace-relative link to each request folder. The convention ensures consistent creation, update, validation, and consumption by both humans and automation.

## Scope
This convention applies to the single file:
- `.aib_memory/requests_register.md`

It governs file **location**, **name**, **schema**, **allowed values**, **sorting**, **editing rules**, **validation rules**, and **automation behavior** for registering AIB requests and their states.

## Deterministic Role in AIB
- The register is the **single source of truth** for resolving the **Active** request when no `request_id` is provided to an action.  
- Exactly **one** request may be in `Active` state at any time; any number may be `Closed`.  
- The register is **created/updated** only by AIB tools and direct human edits following this convention.  
- Script-backed actions reference and update this file as follows:
  - `create-request` **appends** a new row with `state=Active` after verifying there is no other `Active`.
  - `close-request` **updates** the row for the resolved request to `state=Closed` and sets `closed_at`.  
  - Other actions **must not** modify this file.  

## File Location and Name
- **Location:** `.aib_memory/`
- **File name:** `requests_register.md`
- **Path (normalized):** `.aib_memory/requests_register.md`

## File Format
- **Encoding:** UTF-8
- **Markup:** Markdown
- **Structure:** One (and only one) header paragraph (optional short description) followed by **one Markdown table** with a fixed set of columns and no extra tables below it.
- **No versioning or author header**; do not include document-level metadata sections.

### Required Table Schema (in order)
The register table **must** contain exactly these columns and in this order:

1. `request_id` — **Unique**, immutable request identifier with format `R-<YYYYMMDD>-<HHmi>`.
2. `title` — Short, human-readable request title (free text; ≤ 120 chars).
3. `folder` — Workspace-relative path to the request folder:  
   `.aib_memory/requests/<request_id>-<kebab-title>/`
4. `state` — One of `Active` or `Closed`.
5. `created_at` — ISO local timestamp when the request was created, format `YYYY-MM-DD HH:MM:SS ±HHMM` (24h, with seconds and timezone offset).
6. `closed_at` — Local timestamp when the request entered `Closed`; empty if `state=Active`.

> **Notes**
> - Timestamps are local to the operator’s environment at the time of execution.
> - Path separators use forward slashes (`/`) for portability.
> - `folder` path must **exist** after `create-request` completes.

### Example Table Skeleton
```markdown
Request register for AIB requests.

| request_id        | title                         | folder                                                     | state   | created_at        | closed_at          |
| ---               | ---                           | ---                                                        | ---     | ---               | ---                |
| R-YYYYMMDD-HHmi   | <title>                       | .aib_memory/requests/R-YYYYMMDD-HHmi-<kebab-title>/        | Active  | YYYY-MM-DD HH:MI  |                    |
```

### Worked Example Rows
```markdown
| request_id        | title                         | folder                                                                 | state   | created_at        | closed_at          |
| ---               | ---                           | ---                                                                    | ---     | ---               | ---                |
| R-20260307-1342   | create backend layer          | .aib_memory/requests/R-20260307-1342-create-backend-layer/            | Closed  | 2026-03-07 13:42  | 2026-03-07 16:05   |
| R-20260308-0918   | docs: seed references         | .aib_memory/requests/R-20260308-0918-docs-seed-references/            | Active  | 2026-03-08 09:18  |                    |
```

## Naming and Folder Rules
- **Request ID (`request_id`)**
  - Pattern: `^R-[0-9]{8}-[0-9]{4}$` → `R-<YYYYMMDD>-<HHmi>`.
  - Generated by automation; never edited later.
- **Folder Naming**
  - `<request_id>-<kebab-title>/`, where `<kebab-title>` is the lowercased `title` with spaces and punctuation normalized to hyphens; multiple hyphens collapsed to one.
  - Folder lives under `.aib_memory/requests/`.
- **Title Normalization**
  - Human-editable; changing `title` **does not** retroactively rename the folder once created.

## Allowed Values and Constraints
- `state ∈ { Active, Closed }`
- Exactly **one** row with `state=Active` at any time.
- `request_id` is **unique** and **immutable**.
- `folder` is **unique** and must reference a single existing directory.
- `created_at` **required** for all rows.
- `closed_at` **required** when `state=Closed`; **empty** when `state=Active`.
- Table **must be sorted ascending by `request_id`**.

## Editing Rules
- **Human edits are allowed** only for `title` on any row; other columns are automation-owned.
- **Do not** reorder columns, rename columns, or insert auxiliary columns.
- New rows are **appended by automation**; human-manual insertion is discouraged and must follow all rules if done.
- Deleting rows is **forbidden**; requests are immutable once registered. If a request was opened in error, **close** it.

## Validation Rules (Normative)
A validator **MUST** enforce all the following:

1. **Schema integrity**
   - Table has **exactly** the required columns, in order.
   - No duplicate header rows; no additional tables below.
2. **Identifiers**
   - `request_id` matches the regex and is unique.
3. **State model**
   - `state` value is valid.
   - At most one `Active` row.
   - `closed_at` present iff `state=Closed`; empty otherwise.
4. **Paths**
   - `folder` is relative (no drive letters, no leading `../`).
   - `folder` format matches `.aib_memory/requests/<request_id>-<kebab-title>/`.
5. **Timestamps**
   - `created_at` and `closed_at` (when present) match `YYYY-MM-DD HH:MM:SS ±HHMM`.
   - `closed_at >= created_at`.
6. **Sorting**
   - Rows are sorted by `request_id` ascending.
7. **Idempotency**
   - Re-validating without changes yields no diffs or re-writes.

On any validation failure, tools **must** return a human-readable error and **must not** write partial changes.

## Automation Behavior
- **Creation (`create-request`)**
  - Fails if an `Active` row exists.
  - Generates `request_id`, `folder`, `created_at`, sets `state=Active`.
  - Creates the request folder with the seed files:
    - `request.md`, `implementation.md`
- **Closure (`close-request`)**
  - Resolves the target row (explicit `request_id` or the current `Active`).
  - Sets `state=Closed`, sets `closed_at`, preserves other fields.
- **Read resolution**
  - When actions require a request and no explicit `request_id` is provided, resolve the single `Active` row; otherwise, fail if none or more than one found.

## Human Guidance
- Keep `title` concise and informative (≤ 120 chars).
- Do not manually change `request_id`, `folder`, `created_at`, `state`, `closed_at`.

## Minimal Starter Content (for empty register)
Use the skeleton below when the file is first created:

```markdown
Request register for AIB requests.

| request_id | title | folder | state | created_at | closed_at |
| --- | --- | --- | --- | --- | --- |
```

## Change History Expectations
- The register is **append-only** in intent; rows are never deleted.
- Column mutations beyond `title` are automation-owned.
- Renaming the `title` **does not** move the `folder`.

## Rationale and Interoperability
- Single-active-request constraint guarantees deterministic action routing.
- Fixed schema and ordering enable robust parsing across tools and editors.
- Relative paths and forward slashes ensure portability across OS and shells.
