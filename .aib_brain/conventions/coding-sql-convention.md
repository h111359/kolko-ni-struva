# Coding SQL Convention

**Scope:** Normative  
**Applies to:** All SQL script files (`.sql`) created or edited by the AI Automation Agent during any `implement` workflow run, including DDL, DML, stored procedures, views, and functions.  
**Extends:** `coding-general-convention.md` — all rules defined there MUST be applied in addition to the rules below.

---

## 1. Purpose

This convention defines SQL-specific commenting and code-quality rules to ensure that AI-generated SQL code is readable, auditable, and self-documenting for developers and data engineers working with relational or cloud data platform environments.

---

## 2. Scope & Normative Language

This convention applies to:

- All `.sql` files created or modified by the AI Automation Agent.
- SQL embedded in Python, JavaScript, or other host languages when the SQL block is longer than one line.

Out of scope:
- ORM-generated migration files where SQL is auto-generated by a framework tool.
- DAX formulas — see `coding-dax-convention.md`.

Normative keywords **MUST**, **MUST NOT**, **SHALL**, **SHOULD**, and **MAY** are interpreted per BCP 14 (RFC 2119 / RFC 8174).

---

## 3. File-Level Header

Every SQL file MUST begin with a block comment header using `--` line comments.

The header MUST contain:

- The script name and type (e.g., DDL, DML, stored procedure, view).
- A one-sentence description of the script's purpose.
- The target database or schema.
- Any prerequisite objects or dependencies (e.g., tables that must exist before this script runs).

Example:

```sql
-- Script:    create_sales_summary_view.sql
-- Type:      DDL — View creation
-- Purpose:   Creates a denormalized summary view for sales reporting.
-- Target:    analytics_db.reporting schema
-- Requires:  sales.orders, sales.order_items, reference.products
```

---

## 4. Object-Level Description Comment

Every stored procedure, function, or view definition MUST be immediately preceded by a description comment block containing:

- Purpose: what business question or use case this object addresses.
- Parameters (for procedures and functions): name, data type, and meaning of each parameter.
- Return set or output: description of what the procedure returns or modifies.

Example:

```sql
-- Procedure: usp_GetMonthlySalesSummary
-- Purpose:   Returns aggregated monthly sales totals for a given year and region.
-- Parameters:
--   @Year    INT    The calendar year to summarize (e.g., 2025).
--   @Region  NVARCHAR(50)  The sales region filter; NULL returns all regions.
-- Returns:   Result set with columns: Month, Region, TotalSales, OrderCount.
CREATE PROCEDURE usp_GetMonthlySalesSummary
    @Year    INT,
    @Region  NVARCHAR(50) = NULL
```

## 5. Inline Comments for Complex Logic

Inline `--` comments MUST be added to:

- Any subquery or CTE whose purpose is not immediately apparent from its alias.
- Any JOIN condition that uses non-obvious business logic.
- Any CASE expression with more than two branches.
- Any window function (ROW_NUMBER, RANK, LAG, LEAD, etc.).

Comments MUST explain the business or technical reason, not merely restate the SQL.

---

## 6. CTE Comments

Every Common Table Expression (CTE) MUST have a `--` comment immediately above its `WITH` or `, <name> AS` clause that describes:

- What set of data the CTE represents.
- Why it is isolated as a CTE rather than inlined.

---

## 7. Naming Conventions

- Table and view names MUST be in lowercase with underscores as word separators (`snake_case`).
- Stored procedures and functions MUST use a consistent prefix convention (e.g., `usp_`, `ufn_`).
- Aliases MUST be meaningful (not single letters unless they are universally conventional, such as `t` for a single-table query).
- Column aliases in SELECT lists MUST match the business name, not the internal column name when they differ.

---

## 8. Code Quality Rules

- SELECT * is PROHIBITED in production queries; column lists MUST be explicit.
- Implicit JOIN syntax (comma-separated FROM clauses) is PROHIBITED; use explicit INNER JOIN, LEFT JOIN, etc.
- All CAST or CONVERT calls MUST have an inline comment explaining the type conversion reason if it is not obvious.
- NULL-handling logic (ISNULL, COALESCE, NULLIF) MUST be commented with the business rule that justifies the default value.
- Transactions MUST include a comment explaining the atomicity boundary and the rollback condition.
