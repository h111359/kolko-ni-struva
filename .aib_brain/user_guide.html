AI Builder (AIB) — User Guide

A minimal, model-agnostic framework for specification-driven development workflows. All sections expand on click — no network connection required.

AI Builder (AIB) is a minimal but powerful framework for AI specification-based development. AIB serves software development, documentation creation, data processing, and all other activities achievable with AI assistance.

Objectives

  • Analyse user input and generate analysis and plan Markdown files.
  • Generate questions to the user for aspects that could lead to different implementation approaches.
  • Write a detailed plan for action to be performed during the implementation phase.
  • Write implementation details when execution is performed.

Core Philosophy

  • File-first: All state, configuration, and artifacts are Markdown files in the repository — no database, cloud service, or runtime state.
  • Fail-closed: When a required resource is missing or invalid, AIB halts with an explicit error rather than guessing.
  • Determinism: Workflows produce the same output given the same input — every prompt action is reproducible and verifiable.
  • Model-agnostic: Works in VS Code with GitHub Copilot, Claude Code, Cursor, and any other AI coding interface.
Framework location: All reusable AIB assets live in .aib_brain/. All workspace-specific artifacts live in .aib_memory/.
  • Python 3.10+ available in your terminal (python --version to verify).
  • Terminal open at the repository root (where .aib_brain/ and .aib_memory/ are located).
  • An AI coding interface such as VS Code with GitHub Copilot, Claude Code, or Cursor.
No additional libraries required. All AIB tool scripts use only the Python 3.10+ standard library.

Launch the interactive menu from your terminal at the repository root:

Windows

.aib_brain\run.bat

Linux / macOS

sh .aib_brain/run.sh

What the menu shows

  • A state-aware guidance block with your next recommended action based on the current workspace state.
  • When an active request exists, a "Close current request" action appears automatically.
  • Prompt invocation instructions for aib-analyze.md, aib-implement.md, and aib-refresh-context.md.

Workspace states

  • idle — No active request; input.md is empty.
  • input_ready — input.md has content; no active request yet.
  • request_incomplete — Active request exists; analysis not yet run.
  • questions_pending — Q-blocks in input.md await answers.
  • implementation_ready — Plan is ready; implement can be run.
  • amendment_pending — New input detected on an existing active request.
  • unknown — Workspace in an unrecognised state.
  1. Write your intent into the ## Input section of .aib_memory/input.md.
  2. Run analysis in AI chat: 
    Execute .aib_brain/prompts/aib-analyze.md
    (Optional but highly recommended) — auto-creates a request, generates an analysis artifact and a plan, and asks clarifying questions if needed.
  3. Run implement in AI chat: 
    Execute .aib_brain/prompts/aib-implement.md
    If no active request exists, the prompt auto-creates one from input.md and proceeds. The request is closed automatically when implementation completes.

Paste any of the following commands directly into your AI chat interface:

Analyze (plan & research)

Execute .aib_brain/prompts/aib-analyze.md

Reads input.md, auto-creates a request, generates analysis-<id>.md and plan-<id>.md, and produces Q-blocks in input.md when decision points exist.

Implement (execute scope)

Execute .aib_brain/prompts/aib-implement.md

Reads plan-<id>.md, applies all code and documentation changes, creates/appends implementation.md, refreshes context.md, and auto-closes the request.

Refresh Context

Execute .aib_brain/prompts/aib-refresh-context.md

Rebuilds .aib_memory/context.md from the current workspace state. Can be run at any time without an active request.

Full analysis → implement flow (recommended)

  1. Write your request into input.md → ## Input.
  2. Run aib-analyze.md — auto-creates a request, generates analysis-<id>.md and plan-<id>.md.
  3. Review plan-<id>.md and adjust if needed. Answer any Q-blocks in input.md if generated.
  4. Run aib-implement.md — executes scope and closes request automatically.

Direct implement (no formal analysis)

  1. Write your request into input.md → ## Input.
  2. Run aib-implement.md — auto-creates a request from input and implements directly.

Refresh workspace context

  • Run aib-refresh-context.md at any time (no active request required) to update .aib_memory/context.md.

When aib-analyze.md identifies decision points with multiple valid implementation choices where the preferred option has a materially different impact on the codebase, it generates a ## Questions section in input.md.

How it works

  1. After analysis runs, open .aib_memory/input.md. If questions were generated, a ## Questions section appears at the end of the file.
  2. Each question includes:
    • The question text.
    • A > Why this matters: line explaining the implementation impact.
    • Mutually exclusive options with checkboxes; the first option is marked (recommended).
    • A > Answer: field for free-text responses.
  3. Answer questions by checking [x] next to your chosen option, or writing a free-text answer in the > Answer: field.
  4. If you leave a question unanswered, the Answer Application Sub-flow halts with an error message and leaves input.md unchanged. All Q-blocks must be answered before re-running analysis.
  5. Re-run aib-analyze.md — reads your answers, applies them to the relevant plan.md sections, and clears the ## Questions section from input.md.
One Q&A cycle per request: After all questions are answered and analysis re-runs, no new questions are generated (all ambiguities are resolved). The ## Questions section is ephemeral — it is fully cleared after each Q&A cycle.

.aib_memory/instructions.md is a persistent directives file read by every AIB prompt before it executes.

Purpose

Write any workspace-specific rules here — coding conventions, naming rules, always/never behaviors, or references to additional files AIB should read. If absent or empty, all prompts continue normally.

How to use it

  • Open .aib_memory/instructions.md in any text editor.
  • Add free-form Markdown directives. No schema enforcement applies.
  • Directives take effect on the next prompt invocation — no restart required.
  • You may list additional file paths that AIB prompts should read as supplementary context.
Do not store secrets, credentials, or PII in this file. It is VCS-tracked and visible to anyone with repository access.

The workspace uses two top-level folders to separate framework assets from workspace-specific artifacts:

.aib_brain/          ← Reusable framework assets (never modified by tool scripts)
  conventions/       ← Convention files defining document structure and quality gates
  prompts/           ← Prompt files invoked in your AI chat interface
  tools/             ← Python tool scripts for deterministic AIB actions
  README.md          ← Quick-start overview and workspace guide
  user_guide.html    ← Self-contained interactive user guide (browser-viewable)
  run.bat            ← Windows launcher for the interactive menu
  run.sh             ← Linux/macOS launcher for the interactive menu
  vM.m.p             ← SemVer marker file (e.g. v1.3.0)

.aib_memory/         ← Workspace-specific artifacts (project-scoped)
  archives/          ← Archived snapshots of previous .aib_memory/ states
  attachments/       ← Staging folder for supplementary input files
  logs/              ← Tool execution and action logs
  requests/          ← Closed request subfolders (plan, analysis, implementation)
  context.md         ← Auto-generated product knowledge document
  input.md           ← Primary user-to-agent communication file
  instructions.md    ← Persistent workspace-level behavioral directives
  requests_register.md ← Register of all requests with lifecycle state
  vM.m.p             ← Workspace SemVer marker (matched to brain version)

Each request has an ID in the format R-YYYYMMDD-HHmi. While a request is Active, its artifacts live at .aib_memory/ root. On close, they are moved to the request subfolder under .aib_memory/requests/.

Artifact Created by Description
plan-<id>.md aib-analyze.md Request plan and decisions. Active copy lives at .aib_memory/plan-<id>.md; moved to request subfolder after implementation.
analysis-<id>.md aib-analyze.md Reasoning artifact with 5 mandatory sections: Overview, Files Read, Input Interpretation, Research Results, Implementation Alternatives. Not read by implement.
implementation.md aib-implement.md Append-only implementation log. Created from scratch; a new entry is appended on each implementation run.
inputs/input-archive-*.md aib-analyze.md Archived input.md snapshot per analysis run. Never read by any AIB prompt after archiving.
  • ADR — Architecture Decision Record: a document capturing a significant architectural choice, its rationale, and consequences.
  • AI Builder — Minimal, model-agnostic framework for specification-driven development in a repository workspace. Abbreviated AIB.
  • AIB — Abbreviation for AI Builder.
  • Analysis — A reasoning artifact (analysis-<request_id>.md) generated per request; also updates plan-<request_id>.md with Plan and Decisions sections.
  • Auto-close — Behavior whereby aib-implement.md invokes close-request.py automatically after successful implementation.
  • Auto-request — Behavior of aib-analyze.md whereby, when no Active request exists and input.md is non-empty, it auto-creates a request and generates plan.md from input.md content.
  • Brain Folder — Folder (.aib_brain/) containing reusable AIB assets: prompts, conventions, and tool scripts.
  • CI — Continuous Integration: automated pipeline that runs on pull request events.
  • Convention File — A file in .aib_brain/conventions/ that defines the required structure and validation rules for a specific product doc or coding standard.
  • Curated Change Log — logs/next_version_changes.md: append-only Markdown bullet list maintained by the AI agent during implementation; preferred source of Changes: bullets in the next per-version release log; reset to empty by CI after incorporation.
  • FR — Functional Requirement.
  • Implementation Log — Request-scoped on-demand record of changes, tests, and outcomes (implementation.md); created from scratch by aib-implement.md.
  • Input Archive — File written by aib-analyze.md to <request-folder>/inputs/input-archive-<YYYY-MM-DD_HH-MI-SS>.md preserving the original input.md content for audit; MUST NOT be read by any prompt.
  • input.md — Ephemeral primary user-agent communication channel. Seeded by initialize.py; written by the developer; read and processed by aib-analyze.md; reset to seed template after processing.
  • instructions.md — Persistent workspace-level behavioral directives file at .aib_memory/instructions.md. Read by every AIB prompt as the first step before executing its main logic. If absent or empty, prompts continue normally.
  • Memory Folder — Folder (.aib_memory/) containing workspace-specific artifacts: requests, registers, context.md, and input.md.
  • NFR — Non-Functional Requirement.
  • PII — Personally Identifiable Information.
  • Product Doc — A convention-governed documentation file (e.g., context.md); editable by automation when the active request authorises the change.
  • Prompt Action — A .aib_brain/prompts/aib-*.md file that defines the instructions an AI agent executes to produce a specific AIB artifact. Invoked directly in an AI coding interface; model-agnostic.
  • Request — Tracked unit of work with a stable request identifier (R-YYYYMMDD-HHmi) and lifecycle state (Active or Closed).
  • SemVer — Semantic Versioning, following the pattern MAJOR.MINOR.PATCH.
  • SemVer Marker File — A single empty file in .aib_brain/ whose filename encodes vMAJOR.MINOR.PATCH; represents the active product version.
  • SLO — Service Level Objective.
  • Tool Script — A Python script in .aib_brain/tools/ implementing a deterministic AIB action.
  • User Guide — Self-contained HTML file at .aib_brain/user_guide.html; browser-viewable reference documentation for AIB users covering the full AIB workflow, prompt invocations, Q&A mechanism, workspace structure, and glossary. Requires no network connectivity to open.
  • VCS — Version Control System.
  • Versioned Archive — Zip file created by CI per version bump, stored in versions/aib_brain_vX.Y.Z.zip; committed to VCS; used for AIB installation.
  • Workspace — The repository root containing both the AIB brain folder and memory folder plus any product-specific code and configuration.
