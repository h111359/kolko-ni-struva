What is AIB?

AI Builder (AIB) is a minimal, model-agnostic framework that helps software development teams follow a specification-driven workflow when working with AI coding assistants. Instead of issuing freeform instructions directly in a chat window, AIB gives every change a structured lifecycle: from written intent, through analysis and planning, to implementation and closure — with a full audit trail preserved in the repository.

Core ideas

  • Specification-driven: Every change starts with a written request that describes the desired outcome. AIB generates clarifying questions, produces a plan, and executes the implementation in a structured manner.
  • Request: AIB works in cycles of requests. Each request is a single, self-contained change that is analyzed, planned, and implemented. AIB maintains a clear record of all requests in the workspace.
  • Context-aware: AIB maintains a clear understanding of the workspace state, ensuring that each action is informed by the current context. Like a developer, AIB reads the codebase, documentation, and any relevant files and considers their implications before making changes. After the implementation, AIB updates the context.
  • Behavior-Format-Actions: AIB separates the specification of desired behavior in form of prompts from the format of the output in form of conventions, from actions in form of tools.
  • File-first: All state, configuration, and artifacts are plain Markdown or YAML files checked into the repository. No database, cloud service, or runtime process is required.
  • Model-agnostic: AIB works with any AI coding interface — VS Code with GitHub Copilot, Claude Code, Cursor, or any assistant that can read files and follow instructions.
  • Minimal footprint: AIB requires only Python 3 (standard library). No additional packages are needed.
  • Keep-memories: Instead to write prompts directly in the chat which are ephemeral, AIB stores the conversation and decisions in memory files for persistence.

Two key folders

AIB adds exactly two folders to any workspace. Everything else in your repository is untouched. These folders are equivalent to a "brain" and a "memory" for the AI agent, and they are managed separately.

Brain is provided by AIB and should not be edited. It is product agnostic and contains the framework's prompts, tools, and conventions.

Memory is workspace-specific and is editable by the user. Here the communication between the human and the AI agent happens. It contains the workspace's product context, the developer's intent, and the persistent instructions for the AI agent. The memory is created and managed by AIB, but the user can edit some parts following the AIB rules. Memory contains the input.md, context.md, instructions.md, and aib-setup.yaml files.

input.md is the main communication file between AIB and the user. It have multiple purposes:

  • Here stays the current request state and parameters
  • It is the channel where the user describes the change needed to be realized by the current request.
  • AIB generates also questions to the user like a real developer would do when clarifying a request.
The folder .aib_memory/attachments could be populated with additional files related to the current input - similar to the idea of attachments in an email.

context.md is the file where AIB stores the current workspace context. It is automatically generated and updated by AIB after each request so no need to edit it manually. The user still can edit it directly observing the expected format.

instructions.md is the file where the user can write specific for the workspace instructions for AIB. It is a human-editable file. The instructions are read and followed by AIB during its operations.

aib-setup.yaml is the file where the user can configure certain aspects of AIB's behavior or AIB can keep important workspace state information. It is a human-editable YAML file with flat top-level keys: memory_version, default_questions_number, and memory_version_compatibility. The user should not edit the memory_version and memory_version_compatibility keys directly, as they are managed by AIB. 

.aib_brain/      ← Framework assets (prompts, tools, conventions)
                     Provided and maintained by AIB; you do not edit these.

.aib_memory/     ← Workspace-specific artifacts (your requests, context, config)
                     Created and managed per workspace; you configure and use these.
New to AIB? Work through this manual from top to bottom. By the end you will have AIB installed, configured, and ready to process your first request.

Installation

AIB is distributed as a versioned zip archive. You install it by extracting the archive into your workspace root. No package manager, build step, or administrator rights are required.

Prerequisites

  • Python 3.10 or later available in your terminal. Run python --version (Windows) or python3 --version (Linux/macOS) to verify.
  • An AI coding interface — for example VS Code with the GitHub Copilot extension, or any assistant that can open files from your repository.
  • A terminal (PowerShell, Command Prompt, or shell) you can open at the workspace root.
No additional Python packages required. All AIB tool scripts use only the Python 3.10+ standard library. Do not create a virtual environment unless your workspace project requires one for other reasons.

Step 1 — Obtain the archive

Versioned archives are stored in the versions/ folder of the AIB repository, following the naming pattern: 

versions/aib_brain_vX.Y.Z.zip

Download or copy the latest aib_brain_vX.Y.Z.zip file. Choose the highest version number available.

Step 2 — Extract into your workspace

Extract the zip archive so that the .aib_brain/ folder appears at the root of your workspace (the same folder that contains your project files).

Windows
Linux / macOS

Open PowerShell at your workspace root and run:

Expand-Archive -Path "versions\aib_brain_v1.5.2.zip" -DestinationPath "."

Replace v1.5.2 with the version you downloaded. After extraction, confirm that a .aib_brain\ folder now exists in your workspace root.

Open a terminal at your workspace root and run:

unzip versions/aib_brain_v1.5.2.zip -d .

Replace v1.5.2 with the version you downloaded. After extraction, confirm that a .aib_brain/ folder now exists in your workspace root.

Step 3 — Run the launcher

Run the AIB launcher from your terminal at the workspace root. The launcher initialises the .aib_memory/ folder on first run and displays the state-aware menu.

Windows
Linux / macOS 
.aib_brain\run.bat

Run this command every time you want to see the AIB menu, check workspace state, or access maintenance actions. 

sh .aib_brain/run.sh

Run this command every time you want to see the AIB menu, check workspace state, or access maintenance actions.

On first launch, AIB creates the .aib_memory/ folder and seeds the required files. You should see a welcome menu with your next recommended action.

What was created?

After the first launcher run, your workspace will contain:

.aib_brain/              ← Framework assets (extracted from zip)
  conventions/           ← Document structure rules
  prompts/               ← Prompt files you invoke in your AI chat
  tools/                 ← Python tool scripts
  user_guide.html        ← This manual (browser-viewable)
  run.bat / run.sh       ← Launchers

.aib_memory/             ← Created on first launcher run
  aib-setup.yaml         ← Your configuration file
  context.md             ← Workspace product context (auto-generated)
  input.md               ← Where you write your requests
  instructions.md        ← Your persistent workspace directives
Upgrading? If you are extracting a new version of AIB over an existing workspace, run the launcher after extraction. If the brain version does not match your memory version, AIB will offer an automatic upgrade with safe migration steps.

Configuration

AIB has two user-facing configuration files, both located in .aib_memory/. You edit these files directly in any text editor; no restart is required.

aib-setup.yaml

.aib_memory/aib-setup.yaml is a small YAML file that holds AIB's own operational settings. It is generated automatically on first launcher run with sensible defaults. Most users only need to change default_questions_number.

.aib_memory/aib-setup.yaml
memory_version
The version of .aib_memory/ this workspace was initialised with. Do not edit this field manually. AIB manages it automatically during upgrades.
default_questions_number
The minimum number of clarifying questions AIB will generate per analysis run when genuine decision points exist. Default is 5. Increase this if you want AIB to probe your intent more deeply; lower it for concise requests where fewer decision points are expected.
memory_version_compatibility
An upgrade-progress flag managed by AIB. Values: compatible (normal operation), not-compatible (version mismatch), initialized-not-populated (upgrade complete, migration pending). Do not edit this field manually.
Never store secrets, credentials, tokens, or sensitive information in aib-setup.yaml. This file is VCS-tracked and visible to everyone with repository access.

instructions.md

.aib_memory/instructions.md is a free-form Markdown file containing persistent directives that every AIB prompt reads at the start of its execution. Think of it as a standing instruction set you give to the AI agent for this workspace.

When to use it

Use instructions.md to capture any rule or preference that should apply to all AIB runs in this workspace. For example:

  • Coding style or naming conventions specific to this project.
  • Commit message format requirements.
  • A directive to always update a specific documentation file after implementation.
  • References to additional files that AIB should read as supplementary context.

What belongs here

  • Workspace-level rules that apply to every request, not just one.
  • Concise Markdown directives — no schema enforcement applies.
  • Directives take effect on the next prompt invocation immediately.

What does NOT belong here

  • Secrets, credentials, API tokens, passwords, or any sensitive information. This file is VCS-tracked.
  • Request-specific instructions — those belong in input.md ## Input.
  • Deep architectural documentation — use context.md or extension files for that.
Tip: If instructions.md is absent or empty, all AIB prompts continue normally. You do not need to populate it before running your first request.

Your First Request

This section walks you through a complete AIB workflow from start to finish. Follow these steps in order the first time you use AIB in a workspace.

  1. Open the workspace in your AI coding interface.
    Open the repository folder in VS Code (or your preferred AI-capable editor). Confirm that both .aib_brain/ and .aib_memory/ exist at the workspace root.
  2. Launch the AIB menu.
    Open a terminal at the workspace root and run the launcher:

    Windows: .aib_brain\run.bat
    Linux/macOS: sh .aib_brain/run.sh

    The menu shows your current workspace state and the recommended next action.
  3. Write your request into input.md.
    Open .aib_memory/input.md in your editor. Locate the ## Input section and write a clear, bounded description of the change you want AIB to help you implement. Be specific: describe the goal, any constraints, and the files or areas of the codebase involved.
  4. Run analysis.
    In your AI chat interface, type and send: 
    Execute .aib_brain/prompts/aib-analyze.md
    AIB reads input.md, creates a request with a stable ID (R-YYYYMMDD-HHmi), generates an analysis document, and writes a detailed plan file. It may also generate clarifying questions if your request has genuine decision points.
  5. Review the generated analysis and plan.
    After analysis completes, find the generated files in .aib_memory/:
    • analysis-<id>.md — AIB's research and reasoning.
    • plan-<id>.md — The step-by-step implementation plan.
    Read both files. Adjust the plan if needed before proceeding.
  6. Answer any Q-blocks (if generated).
    If AIB generated questions, open .aib_memory/input.md and look for the ## Questions section. Answer each question by checking [x] next to your choice or writing a free-text answer in the > Answer: field. All questions must be answered before proceeding.
  7. Re-run analysis if you answered questions.
    If you answered Q-blocks in the previous step, run analysis again: 
    Execute .aib_brain/prompts/aib-analyze.md
    AIB applies your answers to the plan and clears the questions section.
  8. Run implementation.
    When the plan looks correct and no questions remain, run implementation: 
    Execute .aib_brain/prompts/aib-implement.md
    AIB reads the plan, applies all code and documentation changes, and then automatically closes the request.
  9. Confirm the request is closed.
    After implementation completes, run the launcher again to confirm the workspace is back to idle state. The closed request artifacts are preserved in .aib_memory/requests/<request-folder>/ for audit traceability.
  10. Start the next request.
    Write your next request into .aib_memory/input.md ## Input and repeat from step 4.
Shortcut: If your request is straightforward and you don't need a formal analysis, you can skip directly to implementation. Write your intent in input.md and run: 
Execute .aib_brain/prompts/aib-implement.md
AIB will auto-create a request from your input and implement it directly.

Prompt Reference

AIB actions are triggered by typing a prompt invocation command into your AI chat interface. Each command tells the AI to read and execute the corresponding prompt file. Copy and paste these commands exactly as shown.

Analyze — research and plan a request
Execute .aib_brain/prompts/aib-analyze.md

Reads input.md, auto-creates a request, generates analysis-<id>.md and plan-<id>.md, and writes clarifying Q-blocks to input.md when decision points exist. Use this before implementation for any non-trivial change.

Implement — execute a planned request
Execute .aib_brain/prompts/aib-implement.md

Reads plan-<id>.md, applies all code and documentation changes, refreshes context.md, and auto-closes the request. If no active request exists, it auto-creates one from input.md and proceeds directly.

Modify — direct implementation without analysis
Execute .aib_brain/prompts/aib-modify.md

Reads input.md ## Input as a direct implementation directive and applies changes immediately, without a prior analysis or plan cycle. Use this for simple, well-understood changes where a full analysis cycle is unnecessary.

Refresh Context — rebuild product context
Execute .aib_brain/prompts/aib-refresh-context.md

Rebuilds .aib_memory/context.md from the current workspace state. No active request is required. Run this after significant structural changes to your workspace to keep AIB's product understanding up to date.

Sync Spec — align context with an external specification
Execute .aib_brain/prompts/aib-sync-spec.md

Reads an external specification file (path provided in input.md ## Input), generates Q-blocks for any contradictions with context.md, and adds non-contradictory items as new context entries or [PLANNED] tags.

Input from Context — populate input from planned items
Execute .aib_brain/prompts/aib-input-from-context.md

Reads all [PLANNED] entries and ## Issues from context.md and appends structured goal bullets to input.md ## Input. Useful for turning your context backlog into a ready-to-analyze request. Requires no active request.

Clarify — context-aware clarification pass
Execute .aib_brain/prompts/aib-clarify.md

Performs a context-aware clarification pass on the active request. Before invoking this prompt, use the AIB menu to run Create Clarify Context, then attach both .aib_memory/input.md and the generated context-compilation-<timestamp>.md file to your chat session.

Troubleshooting

If AIB is not behaving as expected, work through the following general checks before seeking further help.

Verify Python is available

AIB requires Python 3.10 or later. Open a terminal and run: 

python --version

On Linux/macOS you may need to use python3 --version. If Python is not found, install it from python.org and ensure it is on your system PATH.

Verify the terminal is at the workspace root

Every AIB launcher and tool command must be run from the directory that contains .aib_brain/ and .aib_memory/. Running from a subfolder will cause tool scripts to fail with a “workspace not found” error.

Verify both folders exist

Confirm that .aib_brain/ and .aib_memory/ are both present at the workspace root. If .aib_memory/ is missing, run the launcher to initialise it: 

.aib_brain\run.bat        (Windows)
sh .aib_brain/run.sh      (Linux/macOS)

Run the launcher to inspect state-aware guidance

The AIB launcher displays a state-aware guidance block with your recommended next action. If you are unsure what to do next, running the launcher is always a safe first step.

Check input.md for pending state

Open .aib_memory/input.md and look at the YAML frontmatter at the top of the file. Check the status field:

  • idle — No active request. Write your intent into ## Input to begin.
  • analysis_ready — A request exists and is ready for implementation.
  • questions_generated — Unanswered Q-blocks exist. Answer them in the ## Questions section before re-running analysis.

Check context.md if AIB misunderstands the workspace

AIB uses .aib_memory/context.md as its primary product knowledge document. If AI responses seem out-of-date or incorrect about your workspace structure, run: 

Execute .aib_brain/prompts/aib-refresh-context.md

This rebuilds context.md from the current workspace state.

Avoid manually editing tool-managed state

The YAML frontmatter at the top of input.md is managed by AIB tool scripts. Do not edit fields such as request_id, status, input_verification_result, or context_verification_result manually unless a recovery procedure explicitly instructs you to do so. Editing these fields incorrectly can put the workspace into an inconsistent state.

If you encounter an unexpected error from a prompt, read the error message carefully. AIB error messages are literal and prescriptive — they tell you exactly what is wrong and what to fix before re-running.

Best Practices

Following these practices will help you get the most out of AIB and avoid common pitfalls.

Write clear, bounded requests

Describe one coherent unit of work per request in input.md ## Input. Include the goal, any constraints, and the files or areas of the codebase involved. Vague or open-ended requests lead to imprecise plans and unexpected implementation choices.

Prefer analysis before implementation for non-trivial changes

For any change that touches multiple files, introduces new abstractions, or has non-obvious trade-offs, run aib-analyze.md first. The analysis and plan files give you a review checkpoint before any code is written.

Answer Q-blocks carefully

Clarifying questions are generated only when AIB identifies genuine decision points where different choices lead to materially different implementations. Your answers directly shape the plan — take the time to read the “Why this matters” context before answering each question.

Keep instructions.md concise and stable

Every AIB prompt reads instructions.md at the start of execution. Keep directives focused and actionable. Avoid adding temporary or request-specific notes here; those belong in input.md.

Never store secrets or sensitive information in AIB artifacts

AIB artifacts — input.md, context.md, instructions.md, aib-setup.yaml, analysis files, plan files — are all plain text files committed to your repository. Never put API keys, passwords, tokens, personal data, or any other sensitive information into these files.

Refresh context after significant workspace changes

context.md is AIB's product knowledge document. If you restructure your codebase, rename major components, or make other large changes outside of AIB requests, run aib-refresh-context.md to keep AIB's understanding current.

Review generated plans before implementation

Always read plan-<id>.md before running implementation. The plan lists every file that will be created or modified and every command that will be run. If anything looks incorrect, update the plan before proceeding — it is a plain Markdown file you can edit directly.

Keep requests scoped to one coherent unit of work

Avoid combining multiple unrelated changes in a single request. AIB is most effective when each request has a single, well-defined goal. Use separate requests for separate concerns.
