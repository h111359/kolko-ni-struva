## Overview

- Request ID: R-20260526-0416

- Request title: Exclude total pages from pagination controls

### Background

The active request asks for the landing-page flat-table pagination to stop calculating the total number of pages and to stop exposing a last-page control. The requested pagination model must still let users go to the first page, the previous page, the next page, and a specific page number.

### Scope

- Update the landing-page flat-table pagination behaviour in `react-app/src/components/LandingPage.jsx` so it no longer depends on total-page calculation.

- Replace the current total-pages indicator and last-page affordance with a direct specific-page navigation affordance that fits the requested control set.

- Update the pagination styling and automated frontend tests that currently assume a five-element bar with a total-pages label.

- Reassess whether the landing-page count fetch in `react-app/src/lib/dataService.js` remains necessary once total-page calculation is removed from this flow.

### Out of scope

- Changes to grouped-result pagination are out of scope because the grouped table currently renders as a non-paginated aggregate view.

- Changes to page size, filter semantics, Supabase schema, or unrelated landing-page visual redesign are out of scope.

- Reworking backend RPC contracts beyond what is strictly required to support the updated flat-table pagination is out of scope.

## Input Interpretation

The requested change applies to the React landing page flat-table browsing experience. The pagination model for detailed result rows must no longer derive or display a total page count, which means the UI must stop depending on the deferred count request that currently feeds the `Стр. X от Y` indicator and the last-page action. The resulting pagination controls must expose only the behaviours explicitly requested by the developer: navigate to the first page, navigate to the previous page, navigate to the next page, and jump directly to a user-specified page number.

This request is limited to user-visible pagination behaviour and its supporting frontend data flow for the flat result table. It does not request new filters, new aggregation behaviour, or a broader redesign of the landing page. The expected implementation should preserve the existing row-first loading model, keep the page usable while rows are loading, and avoid reintroducing a blocking total-count dependency into the main row-fetch path.

## Research Results

### Workspace pattern-scan findings

- `react-app/src/components/LandingPage.jsx` currently treats total-count loading as a secondary asynchronous phase: `loadRows` renders the current page first, then triggers `fetchLandingPageCount(filters)` to populate `totalCount`, `totalPages`, and the final enabled state for the last-page button.

- The current flat-table pagination bar is implemented as five visible elements in `react-app/src/components/LandingPage.jsx`: first, previous, indicator, next, and last. The indicator shows `Стр. X от Y` when the count resolves, or `Стр. X ...` while count resolution is pending.

- `react-app/src/App.css` documents and styles the pagination bar as a five-element layout whose center element is a passive indicator and whose edge buttons include both first and last navigation.

- `react-app/src/components/LandingPage.test.jsx` currently validates the integration by asserting `Стр. 1 от 3` when `fetchLandingPageCount` returns `250`, which couples the test suite to total-page calculation.

- `react-app/src/lib/dataService.js` exposes `fetchLandingPageCount` as a dedicated RPC wrapper for `get_landing_page_count`, and its inline documentation explicitly states that the count exists to support total-page information without slowing the row query.

- `react-app/src/lib/dataService.test.js` has a dedicated unit-test block for `fetchLandingPageCount`, so removing the count from the landing-page UX does not automatically require removing the helper unless no remaining caller needs it.

### Feasibility and acceptance framing

- The request is feasible within the current React architecture because the flat-table pagination state is concentrated inside `react-app/src/components/LandingPage.jsx`, with only one supporting data-service helper tied to total-page calculation.

- The primary behavioural success condition is that users can paginate detailed rows without any `Стр. X от Y` or equivalent total-page output and without any last-page control being rendered.

- A second behavioural success condition is that direct page-number navigation remains possible even though the UI no longer knows the last page in advance.

- A third behavioural success condition is that previous and next controls keep the current boundary behaviour, disabling or preventing invalid backward navigation from the first page and invalid forward navigation when the fetched page does not imply another page.

### Industry knowledge

- GOV.UK Design System guidance for item-list pagination says pagination should be shown only when there is more than one page and that previous or next links should not be shown when users are already at the boundary page. Applicability: this aligns with retaining first, previous, and next controls only when they are meaningful, and it supports the existing pattern of suppressing unusable navigation states rather than forcing a full page-range display.

- GOV.UK guidance also recommends numbered page navigation when users need to move among similar result pages, but the examples depend on a known page range and often include first and last references with ellipses. Applicability: because this request explicitly removes total-page calculation, the repository should not emulate a numbered-range paginator that implies knowledge of the full range; a direct page-number jump is a better fit for the requested constraints.

- Material UI documents first-page and last-page buttons as optional pagination features rather than required controls and separates controlled page selection from the choice of which navigation buttons to render. Applicability: the requested removal of the last-page control is consistent with mainstream component behaviour and does not create a non-standard pagination model by itself.

- Material UI accessibility guidance emphasizes explicit aria labels such as “go to first page”, “go to previous page”, and “go to page N” for pagination items. Applicability: if the current passive indicator becomes an interactive specific-page control, the replacement needs an explicit accessible label in Bulgarian so keyboard and assistive-technology users can understand the action.

- WAI-ARIA Authoring Practices advises preferring native semantics over custom ARIA and identifies spinbutton-style numeric entry as the appropriate pattern when users must enter a discrete numeric value. Applicability: a native numeric input for page jumping is a safer and more accessible fit than a custom pseudo-button control if the implementation chooses free-form page entry.

### AI Agent critique

- `.aib_memory/input.md` states the requested behaviour clearly enough to identify the target surface, but it does not define how a “specific page number” should be entered or submitted, so the implementation will need to keep that interaction minimal and explicit.

- `react-app/src/components/LandingPage.jsx` currently carries two pagination models at once: row-first rendering for responsiveness and deferred total-count fetching for total-page awareness. That split is exactly the kind of coupling this request is trying to remove.

- `react-app/src/components/LandingPage.jsx` hardcodes the total-page string and the last-page button in the render path, so the request cannot be satisfied by styling changes alone.

- `react-app/src/App.css` contains comments that describe the pagination bar as a five-element control with a passive indicator and last-page edge button. Those comments will become inaccurate if the component changes shape.

- `react-app/src/components/LandingPage.test.jsx` verifies only the old total-page presentation. It does not currently cover the requested direct-page navigation behaviour or the absence of a last-page control.

- `react-app/src/lib/dataService.js` documents `fetchLandingPageCount` as the mechanism for total-page calculation. If the landing page stops calculating total pages, the helper may become an unnecessary network round trip for this UX.

- `react-app/src/lib/dataService.test.js` is healthy as a unit-level RPC contract test, but it will no longer prove anything about landing-page pagination once the component stops consuming the count in this flow.

## Decision Register

### Decision Points

#### Decision Point: Total-page dependency for flat-table pagination

- Task or step: Decide whether the flat-table pagination flow should continue to call the deferred count RPC or should navigate without total-page awareness.

- Why alternatives exist: The current implementation uses the count RPC to display `Стр. X от Y` and enable last-page navigation, but the new request explicitly removes both of those user-visible behaviours.

- Alternative A: Keep the deferred count RPC but hide the total-pages output.

- Description: Preserve the existing count fetch internally while removing only the visible total-pages label and last-page button.

- Key trade-offs: This keeps existing boundary data available, but it violates the request to exclude total-page calculation and preserves an avoidable RPC dependency.

- Expected codebase impact: Minimal UI changes in `react-app/src/components/LandingPage.jsx`, with `react-app/src/lib/dataService.js` left in place for the same call path.

- Alternative B: Remove total-page calculation from the flat-table flow and infer forward availability from page-size results.

- Description: Stop calling the count RPC for flat-table pagination and rely on the fetched page length to determine whether another page may exist.

- Key trade-offs: This matches the request and simplifies the data flow, but it means the UI cannot know the final page in advance.

- Expected codebase impact: `react-app/src/components/LandingPage.jsx` changes its pagination state logic, `react-app/src/components/LandingPage.test.jsx` changes its assertions, and `react-app/src/lib/dataService.js` may lose a landing-page caller.

- Tag: resolved-by-user

- Rationale/Resolution: Choose Alternative B because `.aib_memory/input.md` explicitly says, “Exclude calculation of the total number of pages.” That instruction directly rules out continuing the count-based flat-table pagination model.

#### Decision Point: Visible pagination controls

- Task or step: Decide which controls remain visible in the flat-table pagination bar after total-page calculation is removed.

- Why alternatives exist: The current bar has first, previous, indicator, next, and last controls, while the request defines a narrower navigation set.

- Alternative A: Keep the existing first, previous, indicator, next, and last layout.

- Description: Preserve the current five-element structure and adapt only the center text.

- Key trade-offs: This minimizes layout change, but it directly conflicts with the request because it retains last-page navigation and does not add direct page-number access.

- Expected codebase impact: Mostly text-level changes in `react-app/src/components/LandingPage.jsx` and `react-app/src/App.css`.

- Alternative B: Render first, previous, specific-page control, and next, with no last-page control.

- Description: Replace the passive center indicator with an interactive specific-page affordance while preserving first, previous, and next actions.

- Key trade-offs: This satisfies the requested control set, but it requires new interaction handling and accessibility labeling for the page-jump affordance.

- Expected codebase impact: Render and event-handling changes in `react-app/src/components/LandingPage.jsx`, style changes in `react-app/src/App.css`, and new behaviour checks in `react-app/src/components/LandingPage.test.jsx`.

- Tag: resolved-by-user

- Rationale/Resolution: Choose Alternative B because `.aib_memory/input.md` explicitly says, “Pagination to provide possibility for just next page and previous page, first page and specific page number but not last page.” That source resolves which control types must remain and which one must be removed.

#### Decision Point: Specific-page interaction model

- Task or step: Decide how the user provides a specific page number once the passive indicator is replaced.

- Why alternatives exist: The request requires direct navigation to a specific page number, but the current UI has no interactive page-entry control.

- Alternative A: Use a native numeric page input in the center slot.

- Description: Replace the center indicator with a numeric field that accepts a page number and triggers a jump using the existing row-fetch flow.

- Key trade-offs: This works without precomputing the last page and aligns with native accessibility semantics, but it requires validation of invalid or out-of-range user input.

- Expected codebase impact: Add local form state and jump handling in `react-app/src/components/LandingPage.jsx`, adjust center-slot styling in `react-app/src/App.css`, and add interaction tests in `react-app/src/components/LandingPage.test.jsx`.

- Alternative B: Use a dropdown or numbered-button range for page selection.

- Description: Render an explicit list of selectable page numbers instead of a free-form input.

- Key trade-offs: This makes available targets more visible, but it depends on knowing the page range in advance and therefore conflicts with the removal of total-page calculation.

- Expected codebase impact: Introduce page-list rendering logic and a source of last-page knowledge that does not exist in the requested model.

- Tag: resolved-by-user

- Rationale/Resolution: Choose Alternative A because `.aib_memory/input.md` requires navigation to a “specific page number” while also excluding total-page calculation. In this codebase, a native numeric input is the only direct-page mechanism that satisfies both parts of the request without reintroducing page-range precomputation.
