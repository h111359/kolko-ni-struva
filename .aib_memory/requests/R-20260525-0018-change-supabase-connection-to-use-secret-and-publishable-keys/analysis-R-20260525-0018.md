## Overview

**Request ID:** R-20260525-0018

**Request title:** Change Supabase connection to use secret and publishable keys

### Background

Supabase has introduced a new generation of API keys (`sb_publishable_...` and `sb_secret_...`) that replace the legacy JWT-based `anon` and `service_role` keys. The new keys are opaque non-JWT tokens: `sb_publishable_...` is safe to embed in browser bundles and identifies the public client surface; `sb_secret_...` is a server-only elevated-access key that bypasses Row Level Security and must never appear in browser code. Supabase marks the old JWT keys as legacy and recommends migrating.

The product currently uses `VITE_SUPABASE_ANON_KEY` (a legacy JWT anon key) to initialise the React frontend Supabase client, and `DATABASE_URL` + psycopg2 for all backend ETL database access (`src/load_supabase.py`). The developer has completed a Q-block decision cycle (10 questions answered) and the full implementation path is now resolved.

### Scope

- `react-app/src/lib/supabase.js`: rename `VITE_SUPABASE_ANON_KEY` to `VITE_SUPABASE_PUBLISHABLE_KEY`; add a prefix validation guard rejecting `sb_secret_...` keys and JWT-format keys before the Supabase client is instantiated.

- `menu.py`: rename `VITE_SUPABASE_ANON_KEY` to `VITE_SUPABASE_PUBLISHABLE_KEY` in the `action_local_preview()` credential guard (3 occurrences).

- `tests/test_menu.py`: rename all `VITE_SUPABASE_ANON_KEY` to `VITE_SUPABASE_PUBLISHABLE_KEY` in `patch.dict` keys and string assertions; replace `TestViteAnonKeyRole` (JWT-decode test) with a prefix-check test class `TestVitePublishableKeyFormat`.

- `.env.example`: rename `VITE_SUPABASE_ANON_KEY` to `VITE_SUPABASE_PUBLISHABLE_KEY`; add `SUPABASE_SECRET_KEY` placeholder to the server-side section; update acquisition comments; add a Netlify migration note.

- `README.md`: rename all `VITE_SUPABASE_ANON_KEY` to `VITE_SUPABASE_PUBLISHABLE_KEY`; add a Netlify dashboard migration note subsection.

- `.aib_memory/context.md`: update all references to reflect the new variable name, the prefix validation guard, and the addition of `SUPABASE_SECRET_KEY` in `.env.example`.

### Out of scope

- `src/load_supabase.py`: unchanged — backend ETL keeps `DATABASE_URL` + psycopg2 (Q001-A).

- `requirements.txt`: unchanged — `supabase-py` library is not introduced (Q007-B).

- `createClient()` auth options in `react-app/src/lib/supabase.js`: unchanged — default options are correct for this read-only public app (Q008-A).

- Actual Netlify dashboard environment variable update: a manual operator action — documented but not automated (Q004-A).

- Existing developer `.env` files on local machines: operator responsibility during the hard cutover (Q005-A).

- `src/deploy_netlify.py`: reads only `NETLIFY_AUTH_TOKEN` and `NETLIFY_SITE_ID` — no `VITE_` key references; unaffected.

- PostgreSQL schema, RPC functions, and indexes provisioned by `load_supabase.py`: unchanged.

## Input Interpretation

All ten decision points were raised in the prior Q-block cycle and have been answered by the developer. The resolved implementation path is a frontend-only hard cutover.

The developer requests renaming the React app Supabase client-side key environment variable from `VITE_SUPABASE_ANON_KEY` to `VITE_SUPABASE_PUBLISHABLE_KEY` across all code, tests, documentation, and configuration files in a single hard cutover with no backwards-compatibility fallback. The backend ETL (`src/load_supabase.py`) is unchanged: it continues to use a direct PostgreSQL connection via `DATABASE_URL` + psycopg2, and the new `sb_secret_...` key is not wired into any Python code in this migration.

The `react-app/src/lib/supabase.js` singleton gains a format validation guard: after the empty-check, keys starting with `sb_secret_` or matching the three-segment JWT pattern are rejected with a descriptive `credentialsError` message so that accidentally bundling a server-side key into the browser produces an explicit error rather than silently establishing an unsafe Supabase client.

`tests/test_menu.py` replaces the JWT-decoding `TestViteAnonKeyRole` security test with a prefix-check test that asserts the key value in `.env` starts with `sb_publishable_` and fails if it starts with `sb_secret_`.

`.env.example` documents `SUPABASE_SECRET_KEY` (single placeholder, `sb_secret_...` format) in the server-side section for future REST API use, and the Netlify migration note instructs the operator to rename the dashboard environment variable before deploying the updated code.

**`src/deploy_netlify.py` confirmed unaffected**: references only `NETLIFY_AUTH_TOKEN` and `NETLIFY_SITE_ID`.

**`react-app/vite.config.js`**: `envDir: '../'` causes Vite to read `.env` from the project root. This configuration is unchanged; only the variable name changes. No build configuration modifications are needed.

**`@supabase/supabase-js` v2 compatibility**: `createClient(url, key)` accepts both legacy JWT keys and the new `sb_publishable_...` opaque keys. The client library itself does not enforce key format. The validation guard being added to `supabase.js` is an application-layer check, not a library requirement.

**Test coverage scope for `TestActionLocalPreviewCredentials`**: the three test methods patch `VITE_SUPABASE_ANON_KEY` directly in `os.environ` and assert the error message contains the variable name. All three will fail immediately after the rename unless updated. These tests are in scope.

**`TestViteAnonKeyRole` JWT decoding**: the existing test splits the key on `.` into 3 segments and Base64url-decodes the second segment to extract `role`. New `sb_publishable_...` keys are not JWTs — they do not have three dot-separated Base64url segments. The test would `self.fail()` (not skip) on a new key. The replacement prefix-check test handles both legacy JWT keys (detected via 3-segment structure, skipped gracefully during transition) and new `sb_publishable_...` keys.

**Sequencing risk (hard cutover)**: Netlify injects `VITE_*` variables at build time. If the Netlify dashboard still has `VITE_SUPABASE_ANON_KEY` set when the updated code is deployed, the bundle will contain `undefined` for `VITE_SUPABASE_PUBLISHABLE_KEY`, causing the React app to show only the `credentialsError` screen. The recommended sequencing is: update the Netlify dashboard first, then push the code change. This must be documented explicitly.

### Industry Knowledge

**Finding 1 — Supabase publishable / secret key specification (Supabase documentation)**

Supabase defines `sb_publishable_...` as the public-client key and `sb_secret_...` as the server-side key. The `sb_publishable_...` key maps to the `anon` Postgres role and is governed by Row Level Security; the `sb_secret_...` key maps to the `service_role` Postgres role and bypasses RLS. `@supabase/supabase-js` v2 accepts both legacy JWTs and new opaque keys for `createClient(url, key)`. The existing `GRANT EXECUTE TO anon` on RPC functions in `src/load_supabase.py` continues to be the correct grant for the `sb_publishable_...` key — no DDL changes are needed.

Applicability: confirms the frontend-only migration scope.

**Finding 2 — Vite build-time environment variable injection (Vite documentation, 12-Factor App)**

Vite replaces `import.meta.env.VITE_*` references with their literal values at build time. If a `VITE_*` variable is absent from the build environment, it is replaced with `undefined`. This means renaming a variable in code without updating the build environment (Netlify site settings, local `.env`) silently produces a broken bundle — there is no runtime warning. The `credentialsError` guard in `supabase.js` is the only mechanism that surfaces this misconfiguration to the operator or end-user.

Applicability: motivates the Netlify migration note (Q004-A) and the prefix validation guard (Q010-A).

**Finding 3 — Defense-in-depth for client-side API key format validation (OWASP API Security Top 10, OWASP ASVS)**

Validating the format of a credential at the point of use (fail-fast) is a security best practice that prevents misconfiguration from being silently deployed. Checking that a client-side key does not match the server-side format (`sb_secret_...` prefix or 3-segment JWT structure) ensures that a copy-paste error cannot silently bundle an elevated-privilege key into the browser. The `credentialsError` output path already exists in `supabase.js`; extending it with a prefix guard reuses the existing error-surface mechanism without new infrastructure.

Applicability: directly implements Q010-A.

**Finding 4 — Hard cutover sequencing for env var renames (Google SRE, GitLab configuration management)**

A hard cutover is appropriate when all environments are under the operator's control and the deployment can be coordinated. The recommended sequencing is: (1) update the Netlify dashboard variable first, (2) then push the code change. This means the production environment is ready for the new variable before the code that reads it is deployed, eliminating the broken-bundle window entirely.

Applicability: supports Q005-A; provides the sequencing recommendation captured in the README migration note.

### AI Agent Critique

- `tests/test_menu.py::TestViteAnonKeyRole` reads `.env` using a line-by-line string split (`stripped.split("=", 1)[1]`). This parser does not handle quoted values, export prefixes, or multi-line values. The fragility is pre-existing and not introduced by this migration. The replacement test should inherit the same simple parser for consistency — fixing it is out of scope.

- `tests/test_menu.py` imports `base64` and `json` at the top level for the JWT-decoding security test. After `TestViteAnonKeyRole` is replaced, these imports may become unused. The replacement procedure must check and remove them if no other test uses them.

- `react-app/src/lib/supabase.js` error message is in Bulgarian (`Липсват Supabase credentials...`) while the surrounding code is in English. The prefix validation guard error message should use Bulgarian to match the existing language convention and avoid mixed-language output on the same UI surface.

- `menu.py::action_local_preview()` prints an example block that hardcodes `VITE_SUPABASE_ANON_KEY=<your-anon-key>`. If not updated, a developer following this error output after migration will add the wrong variable to their `.env`.

- `.env.example` currently uses "Supabase anonymous public key (public-safe, not a secret)" as the comment. The replacement comment should use "publishable key" terminology and remove "anonymous" to align with Supabase's new vocabulary.

- `README.md` line 466 security note reads "Client-side credentials (`VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`) use Supabase's public anon key, which is safe to expose in the browser." After migration, the parenthetical, the phrase "anon key", and the description should all be updated for accuracy.

- The Netlify migration note should explicitly recommend updating the Netlify dashboard **before** pushing the code change. Both `.env.example` and `README.md` notes should include this sequencing guidance.

- No issues found that would block implementation.
