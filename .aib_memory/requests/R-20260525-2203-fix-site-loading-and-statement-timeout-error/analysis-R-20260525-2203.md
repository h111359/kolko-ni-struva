# Analysis: R-20260525-2203

**Request ID:** R-20260525-2203  
**Request Title:** Fix site loading and statement timeout error

---

## Overview

### Background

The React app stopped loading locally after the R-20260525-1400 total redesign introduced the unified `LandingPage.jsx` backed by seven new Supabase RPC functions. On mount, the landing page fires six concurrent Supabase RPC calls: five option-list RPCs (`get_lp_options_*`) plus one paginated-rows RPC (`get_landing_page_rows`). The paginated-rows RPC embeds a `COUNT(*) OVER()` window function that forces a full-result-set scan across all matching rows of `fact_prices_lookback` before returning even one page. On a 1.18 million-row table with five JOIN operations, this consistently exceeds the Supabase anon-role default `statement_timeout` of approximately 8 seconds, surfacing as `fetchLandingPageRows: canceling statement due to statement timeout`. The concurrent I/O pressure from the window-function query also starves the five option RPCs of resources, causing all five `Promise.all`-wrapped calls in `loadOptions` to time out together, which leaves every dropdown empty.

### Scope

- `src/load_supabase.py` — modify `get_landing_page_rows` RPC to remove the `COUNT(*) OVER()` window function; add a new `get_landing_page_count` RPC; add `GRANT` for the new RPC; re-provision all RPCs by running the script against Supabase.
- `react-app/src/lib/dataService.js` — update `fetchLandingPageRows` to return rows without `total_count`; add `fetchLandingPageCount` function.
- `react-app/src/components/LandingPage.jsx` — update `loadRows` to call `fetchLandingPageCount` separately; handle async count loading gracefully so pagination renders with best-available count.
- `react-app/src/components/LandingPage.test.jsx` — update mock and test expectations to match new function signatures.
- `react-app/src/lib/dataService.test.js` — add tests for `fetchLandingPageCount`.
- `.aib_memory/context.md` — record the architectural change.

### Out of scope

- Changes to any other RPC function (`get_landing_page_grouped`, `get_lp_options_*`, `get_available_dates`).
- Changes to the ETL pipeline (`src/extract.py`, `src/transform.py`).
- Database index additions beyond what already exists.
- Any changes to authentication, key handling, or `supabase.js`.
- UI redesign or UX changes beyond the minimal adjustment required for async count.

## Input Interpretation

The developer reports that the unified `LandingPage` introduced in R-20260525-1400 fails to load locally. The browser shows an error originating from the `fetchLandingPageRows` function, and the message is `canceling statement due to statement timeout` — a PostgreSQL/PostgREST signal that the Supabase backend aborted the query before it completed. As a secondary symptom, all five cross-filter dropdown lists (date, settlement, category, company, store) render empty. The developer requires the application to load and operate correctly before the request is considered closed.

## Research Results

### Workspace Pattern-Scan Findings

**Root cause — `COUNT(*) OVER()` window function in `get_landing_page_rows`:**  
`src/load_supabase.py` (lines 598–660) defines `get_landing_page_rows` as a `LANGUAGE sql STABLE` function that wraps the paginated query in `SELECT COALESCE(json_agg(row_to_json(t)), '[]') FROM (SELECT ..., COUNT(*) OVER() AS total_count ... OFFSET p_offset LIMIT p_limit) t`. The window function `COUNT(*) OVER()` is computed over the full result set before any row is returned to the caller. Even though `OFFSET` and `LIMIT` restrict the final output to `p_limit` rows, PostgreSQL must first materialize all matching rows (after JOINs and WHERE predicates) to evaluate the window. With `fact_prices_lookback` at 1,185,556 rows and five dimension JOINs, the query cannot complete within the Supabase anon-role `statement_timeout` (approximately 8 seconds).

**Secondary cause — `Promise.all` in `loadOptions` fails on any single timeout:**  
`react-app/src/components/LandingPage.jsx` (lines approx. 100–110) wraps five `fetchLandingPageOptions` calls in a single `Promise.all`. If any one of the five RPC calls fails (including due to statement timeout caused by I/O resource starvation from the concurrent rows query), the `catch` block executes and `setAvailableOptions` is never called — leaving all dropdowns in their initial empty state.

**Index coverage is adequate:**  
`src/load_supabase.py` (lines 193–209) provisions `idx_fact_prices_lookback_date_key`, `idx_fact_prices_lookback_date_store`, `idx_fpl_date_cat`, and `idx_fpl_date_store_category`. These indexes allow the WHERE predicates in the option RPCs to use index scans. The option RPCs themselves are not inherently slow; their failure is a secondary effect of I/O starvation from the rows query.

**Initial state — `selectedDate` is populated before queries fire:**  
`App.jsx` renders `<LandingPage dimensions={dimensions} />` only after `fetchDimensions()` resolves. `LandingPage.jsx` initialises `selectedDate` from `dimensions.dates[0]?.date_key`, so the initial `buildFilters()` call includes a non-null `dateKey`. The queries are not running on a fully unfiltered table; they filter by `date_key` from the first render. The timeout is caused by the window function overhead within a filtered-but-still-large result set, not by a missing date filter.

**Test interface establishes `{ rows, totalCount }` shape:**  
`react-app/src/components/LandingPage.test.jsx` (lines 40–51) defines `makeRowResult()` returning `{ rows: [...], totalCount: 1 }`, and multiple test cases assert on `totalCount`. The `fetchLandingPageRows` function contract in tests is `Promise<{ rows, totalCount }>`. Any fix must preserve this interface or update every test that asserts on it.

**`dataService.test.js` does not test `fetchLandingPageRows` at the RPC level:**  
`react-app/src/lib/dataService.test.js` does not directly test `fetchLandingPageRows` against the RPC signature; it mocks Supabase. A new `fetchLandingPageCount` function will need a corresponding test.

**`get_landing_page_rows` is `LANGUAGE sql STABLE`:**  
SQL-language functions cannot execute `SET LOCAL statement_timeout` (that requires PL/pgSQL). Changing the timeout at the function level would require converting to PL/pgSQL with `SECURITY DEFINER`, which bypasses the anon-role timeout globally. This approach creates a security surface and is not the preferred fix for a performance regression.

### Industry Knowledge

**1. Separate count from paginated data (PostgreSQL pagination best practices):**  
The PostgreSQL community and PostgREST documentation consistently recommend issuing a `SELECT COUNT(*)` as a separate query rather than embedding `COUNT(*) OVER()` in a paginated query. The window function requires the query executor to build the full result set before streaming any output row; by contrast, a separate COUNT query can use an index-only scan and completes independently, allowing the page query to return fast with O(p_limit) work. This is directly applicable: splitting the single RPC into `get_landing_page_rows` (rows only) and `get_landing_page_count` (count only) eliminates the window-function bottleneck without changing the public API contract beyond adding a function.

**2. React concurrent fetch + graceful degradation:**  
The React patterns community (React Query documentation, SWR, and related OSS) recommends firing count and data queries in parallel (via `Promise.all` or equivalent) and rendering the page data as soon as it arrives, with the count field populated once the count query resolves. This avoids waterfall latency. LandingPage can set `totalCount` to `0` initially (showing no total pages) and update it when the count RPC resolves — degrading gracefully even if the count query is slow.

**3. `Promise.all` failure isolation for independent operations:**  
Industry-standard practice (MDN, multiple OSS React component libraries) recommends using `Promise.allSettled` instead of `Promise.all` when the operations are independent and partial success is acceptable. Applied to `loadOptions`, this would allow 4 of 5 option RPCs to succeed even if one fails, populating all but the failing dimension's dropdown. This reduces the blast radius of a single RPC failure without architectural change.

### AI Agent Critique

- **`get_landing_page_rows` mixes pagination concern with aggregation concern:** The window function `COUNT(*) OVER()` is an aggregation that belongs in a separate query. Embedding it into the paginated data query forces a sequential dependency between counting and fetching, violating the separation-of-concerns principle and creating a performance trap.

- **`loadOptions` uses `Promise.all` for independent dimension queries:** If one option RPC fails (for any reason), all five option dropdowns silently go empty with only a `console.error`. Replacing with `Promise.allSettled` and handling per-dimension errors individually would make the component more resilient.

- **`loadOptions` catch block does not call `setError`:** A `loadOptions` failure produces a `console.error` only, giving the end user no visible feedback about why dropdowns are empty. An `setError` call or at minimum a user-visible warning would improve observability.

- **`LandingPage.jsx` initial `selectedDate` assumes `dimensions.dates` is non-empty:** `useState(() => dimensions.dates[0]?.date_key ?? null)` silently defaults to `null` if dim_date is empty. No guard or user-visible error is shown when `dimensions.dates.length === 0`.

- **`get_landing_page_rows` is `LANGUAGE sql STABLE` but performs a `COUNT(*) OVER()` on a large table:** A STABLE function is advertised to the planner as not modifying the database and potentially returning the same value for the same input within a transaction. The window function adds a hidden cost that the planner cannot predict from the function signature alone.

- **No `SET LOCAL statement_timeout` or RPC-level timeout guard:** The anon role's default statement_timeout is never overridden at the function level. For queries on large tables, there is no mechanism to either increase tolerance or fail fast with a clear message.

- **`LandingPage.test.jsx` `makeRowResult` hardcodes `total_count: 1` in the row object and `totalCount: 1` in the wrapper:** This is an implementation detail of how the count was derived (from the window function). After the fix, `total_count` will no longer be present in individual row objects. Tests must be updated.

## Decision Register

### Decision Points

#### Decision Point: Pagination Total Count Strategy

**Context:** `get_landing_page_rows` currently returns `total_count` via a window function embedded in each row. This must be removed to fix the timeout. Once removed, the React app loses access to total page count for pagination rendering. A replacement strategy is required.

**Chosen: Alternative A — Separate `get_landing_page_count` RPC:**  
A new `get_landing_page_count(...)` function with identical filter parameters executes `SELECT COUNT(*) FROM fact_prices_lookback JOIN ... WHERE ...` and returns an integer. The React app calls `fetchLandingPageRows` and `fetchLandingPageCount` in parallel; `totalCount` state is populated asynchronously. Pagination renders with `totalCount = 0` until the count resolves, then updates.
- Benefits: preserves exact total count; minimal React change; count query is independent and faster than window.
- Codebase impact: new RPC in `src/load_supabase.py`; new `fetchLandingPageCount` in `dataService.js`; small change to `LandingPage.jsx` `loadRows`; new test in `dataService.test.js`.

- **Tag:** `resolved-by-user`
- **Resolution:** Developer selected Option A (Q001) — add a separate `get_landing_page_count` RPC.

#### Decision Point: Options Loading Error Handling

**Context:** `loadOptions` wraps five RPCs in `Promise.all`. One failure causes all five dropdowns to go empty. This behaviour is present regardless of the root-cause fix; if the root cause is fixed, the option RPCs are unlikely to fail in normal operation. The question is whether to harden `loadOptions` against partial failures as part of this request.

**Chosen: Alternative A — Keep `Promise.all`; rely on root-cause fix:**  
No change to `loadOptions` error handling. Once `get_landing_page_rows` no longer monopolises I/O, option RPCs complete within timeout. Simpler change set.
- Benefits: minimal scope; fewer files changed.
- Codebase impact: none.

- **Tag:** `resolved-by-user`
- **Resolution:** Developer selected Option A (Q002) — keep `Promise.all`; no change to `loadOptions` error handling.
