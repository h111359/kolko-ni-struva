## Goal

Fix the statement-timeout error that prevents the `LandingPage` from loading. Remove the `COUNT(*) OVER()` window function from `get_landing_page_rows`, introduce a separate `get_landing_page_count` RPC, update the React data service and component to call both in parallel, update all affected tests, re-provision the RPCs in Supabase, and confirm the site loads and tests pass.

## Constraints

- The `fact_prices_lookback` table has approximately 1.18 million rows; queries must avoid full-table materialization without a `date_key` filter unless the user explicitly clears the date filter.
- The Supabase anon role `statement_timeout` is approximately 8 seconds and cannot be changed from the client or from `LANGUAGE sql STABLE` functions without `SECURITY DEFINER` (which is not allowed per this request's out-of-scope boundary).
- No changes to `get_landing_page_grouped`, `get_lp_options_*`, `get_available_dates`, or `supabase.js`.
- No changes to the ETL pipeline files (`src/extract.py`, `src/transform.py`).
- No new database indexes.
- No new Python virtual environment; no new third-party library installations.
- Tests are added in the request's test files, not in a separate folder.
- `Promise.all` in `loadOptions` is left unchanged (Decision Point 2 resolved as Option A).

## Success criteria

- Running `npm test` in `react-app/` passes all existing and new tests with zero failures.
- `fetchLandingPageRows` no longer includes `total_count` in each row object.
- `fetchLandingPageCount` exists in `dataService.js` and returns a plain integer count.
- `LandingPage.jsx` calls `fetchLandingPageCount` and `fetchLandingPageRows` in parallel on every `loadRows` invocation.
- `get_landing_page_rows` SQL in `src/load_supabase.py` contains no `COUNT(*) OVER()`.
- `get_landing_page_count` SQL exists in `src/load_supabase.py` with a `GRANT EXECUTE ... TO anon` statement.
- `dataService.test.js` contains at least one test for `fetchLandingPageCount`.
- `LandingPage.test.jsx` mocks for `fetchLandingPageRows` no longer include `total_count` per row; a mock for `fetchLandingPageCount` is added.

## Plan

### Task 1: Remove `COUNT(*) OVER()` from `get_landing_page_rows` and add `get_landing_page_count` RPC

#### Intent
Rewrite the `get_landing_page_rows` SQL function in `src/load_supabase.py` to return rows only (no window count), and add a new `get_landing_page_count` function with identical filter parameters that returns a plain `BIGINT`.

#### Outputs
- `src/load_supabase.py` — modified `get_landing_page_rows` DDL; new `get_landing_page_count` DDL; new `GRANT` statement.

#### Procedure
Open `src/load_supabase.py` and locate the `CREATE OR REPLACE FUNCTION get_landing_page_rows` block (around line 598). Remove the `COUNT(*) OVER() AS total_count` expression from the inner SELECT list. The function still wraps the result in `json_agg(row_to_json(t))`.

After the closing `$$;` of `get_landing_page_rows` and before the `get_landing_page_grouped` block, insert a new `CREATE OR REPLACE FUNCTION get_landing_page_count` function. The function signature must match `get_landing_page_rows` except the last two parameters (`p_offset INT`, `p_limit INT`) are omitted. The body executes `SELECT COUNT(*) FROM fact_prices_lookback f JOIN ... WHERE (same predicates as get_landing_page_rows)` and returns `BIGINT`.

After all `GRANT` statements for `get_landing_page_rows` and `get_landing_page_grouped`, add:
`GRANT EXECUTE ON FUNCTION get_landing_page_count(INT, INT, INT, INT, INT, TEXT, NUMERIC, NUMERIC) TO anon;`

#### Done criteria
- `get_landing_page_rows` DDL in `src/load_supabase.py` contains no `COUNT(*) OVER()`.
- `get_landing_page_count` DDL exists with correct parameter list and `RETURNS BIGINT`.
- A `GRANT EXECUTE ON FUNCTION get_landing_page_count(...) TO anon` statement is present.

#### Dependencies
None.

#### Risk notes
- If the Supabase project has not been re-provisioned after R-20260525-1400, the old function signature may differ. Running `load_supabase.py` re-provisions with `CREATE OR REPLACE`, which is idempotent.
- The COUNT query on a fully unfiltered `fact_prices_lookback` (no `date_key` filter) will itself scan the full table and may also timeout. This is acceptable because the initial render always has a `date_key` filter (from `dimensions.dates[0]`); the risk is only present when the user explicitly clears the date filter.

### Task 3: Update `LandingPage.jsx` to call `fetchLandingPageCount` in parallel

#### Intent
Update `loadRows` in `react-app/src/components/LandingPage.jsx` to call `fetchLandingPageRows` and `fetchLandingPageCount` in parallel, populating `totalCount` from the count RPC result.

#### Outputs
- `react-app/src/components/LandingPage.jsx` — updated `loadRows` function; updated import line.

#### Procedure
Open `react-app/src/components/LandingPage.jsx`. In the import block at the top of the file, add `fetchLandingPageCount` to the named imports from `'../lib/dataService'`.

Locate the `async function loadRows(filters, page)`. Replace the body so that `fetchLandingPageRows` and `fetchLandingPageCount` are called in parallel using `Promise.all`. Set `rows` from the rows result and `totalCount` from the count result. Keep the existing `setLoading`, `setError`, `setRows`, `setTotalCount`, `setCurrentPage` calls. On error from either call, set the error message, clear rows, and set `totalCount` to `0`.

The updated `loadRows` must produce the same observable state transitions as before: `loading = true` at start, `loading = false` at end (success or error), `rows` and `totalCount` updated atomically after both calls resolve.

#### Done criteria
- `fetchLandingPageCount` is imported in `LandingPage.jsx`.
- `loadRows` calls both `fetchLandingPageRows` and `fetchLandingPageCount` for every invocation.
- `totalCount` state is set from the count RPC result, not from `rows[0].total_count`.

#### Dependencies
Task 2 (the `fetchLandingPageCount` export must exist before the import resolves).

#### Risk notes
- `Promise.all` means a failure in either call (rows or count) triggers the catch block. If the count query is slow and the rows arrive fast, the user will see the loading spinner until both resolve. This is acceptable per the decision: keep `Promise.all` strategy.
