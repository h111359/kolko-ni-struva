## Goal

Eliminate the remaining landing-page `fetchLandingPageRows` statement-timeout failure while preserving the current landing-page browsing UX. The implementation should refactor the current-page row retrieval path so the database resolves a thinner visible-page slice first and only then enriches the returned rows needed by the UI.

## Constraints

- Preserve the existing landing-page UX: no mandatory extra filters and no narrowed browsing flow.
- Keep the React app on the current direct-to-Supabase architecture described in `.aib_memory/context.md`.
- Limit code changes to the landing-page SQL/query path, its frontend contract, related tests, and affected documentation.
- Maintain compatibility with the existing `fetchLandingPageRows` consumer flow in `react-app/src/components/LandingPage.jsx`, adjusting the contract only where the refactor requires it.
- Update stale documentation/comments that currently describe parallel row/count fetching.

## Success criteria

- The landing page can load the current page of flat rows without triggering `fetchLandingPageRows: canceling statement due to statement timeout` for the previously failing use case.
- `src/load_supabase.py` provisions a thinner landing-page row retrieval path aligned with the chosen query-refactor strategy.
- `react-app/src/components/LandingPage.jsx` and `react-app/src/lib/dataService.js` consume the refactored row path correctly and preserve existing browsing behavior.
- Landing-page unit tests and data-service tests pass after the contract changes.
- `.aib_memory/context.md` is updated to reflect the final landing-page fetch behavior and any backend-query changes.
