## Goal

Replace the current five-page React Analytics App with a single unified landing page (`LandingPage.jsx`) that displays `fact_prices_lookback` rows for the most recently loaded date via a server-side PostgreSQL RPC, provides five fully cross-filtered dropdown filters, a product name text filter, two price range inputs, and a two-level grouping mode with avg/min/max aggregations. Remove all four non-Files pages and their components. Add seven new PostgreSQL RPC functions to `src/load_supabase.py`. Update `dataService.js`, `App.jsx`, and `App.css`. Delete all obsolete components and their test files. Update `context.md`.

## Constraints

- The React app is a client-only static SPA deployed on Netlify (no backend, no serverless functions); all new data operations must be Supabase RPC calls using the anon key only.
- Supabase RLS must allow public SELECT on all star-schema tables and EXECUTE on all seven new RPC functions; the operator must run `python src/load_supabase.py` to provision the new RPCs before the landing page is functional.
- The `get_landing_page_grouped` RPC uses dynamic SQL (`EXECUTE format(...)`); grouping column names must be validated against a whitelist before interpolation to prevent SQL injection.
- The `dimensions.files` cache (Map<file_key, {file_name, zip_date}>) is available in `_dims` from `fetchDimensions()` and covers the "Файл" column enrichment if needed client-side; the RPC should resolve `file_name` via a SQL JOIN.
- `queryLog.js` infrastructure and the `backend_sql_audit_log` table are out of scope; retain existing query logging call sites.
- Price range inputs must use `<input type="text">` with regex `/^\d+([.,]\d{1,2})?$/` to handle both `.` and `,` decimal separators (Bulgarian locale).
- B-tree indexes on `fact_prices_lookback(date_key)`, `(settlement_key)`, `(category_key)`, `(store_key)`, `(file_key)` are assumed to exist (context.md A9); verify in Supabase before implementation and raise a blocker if any are missing.
- No changes to `src/extract.py`, `src/transform.py`, or `menu.py`.
- `src/load_supabase.py` changes are limited to appending new RPC DDL blocks to the existing `_CREATE_RPC_FUNCTIONS` string.

### Task 3: Author and provision two main landing-page RPCs

#### Intent
Add `get_landing_page_rows` and `get_landing_page_grouped` PostgreSQL functions to `src/load_supabase.py` to serve the flat detail view (with pagination) and the grouped aggregation view.

#### Outputs
- `src/load_supabase.py` — two additional DDL blocks appended after the Task 2 DDL blocks

#### Procedure
Open `src/load_supabase.py` and append to `_CREATE_RPC_FUNCTIONS`:

RPC A — `get_landing_page_rows(p_date_key INT, p_settlement_key INT, p_category_key INT, p_company_key INT, p_store_key INT, p_product_name TEXT, p_price_min NUMERIC, p_price_max NUMERIC, p_offset INT, p_limit INT)` returns `JSON`. Body logic:
- JOIN `fact_prices_lookback f` with `dim_product dp`, `dim_category dc`, `dim_settlement ds`, `dim_store dstore`, `dim_company dcomp`, `dim_file df` on their respective key columns.
- WHERE clauses: `(p_date_key IS NULL OR f.date_key = p_date_key)`, `(p_settlement_key IS NULL OR ds.settlement_key = p_settlement_key)`, `(p_category_key IS NULL OR f.category_key = p_category_key)`, `(p_company_key IS NULL OR dcomp.company_key = p_company_key)`, `(p_store_key IS NULL OR f.store_key = p_store_key)`, `(p_product_name IS NULL OR dp.product_name ILIKE '%' || p_product_name || '%')`.
- Effective price column: `LEAST(f.retail_price, NULLIF(f.promo_price, 0))` COALESCE'd with `f.retail_price`.
- Price range: `(p_price_min IS NULL OR COALESCE(LEAST(f.retail_price, NULLIF(f.promo_price, 0)), f.retail_price) >= p_price_min)` and `(p_price_max IS NULL OR COALESCE(LEAST(f.retail_price, NULLIF(f.promo_price, 0)), f.retail_price) <= p_price_max)`.
- SELECT columns returned in each row: `file_name`, `product_name`, `category_name` (from dc.name), `settlement_name` (from ds.name), `store_name` (from dstore.store_name), `company_name` (from dcomp.name), `retail_price`, `promo_price`, effective `price` (Цена), `total_count` (window count using `COUNT(*) OVER()`).
- Apply `OFFSET p_offset LIMIT p_limit`.
- Return `json_agg(row_to_json(t))` of the result set.

RPC B — `get_landing_page_grouped(p_date_key INT, p_settlement_key INT, p_category_key INT, p_company_key INT, p_store_key INT, p_product_name TEXT, p_price_min NUMERIC, p_price_max NUMERIC, p_group_by_1 TEXT, p_group_by_2 TEXT)` returns `JSON`. Body logic:
- Validate `p_group_by_1` and `p_group_by_2` against whitelist `ARRAY['settlement_name','category_name','company_name','store_name','date_key']`. If either value is not in the whitelist, RAISE EXCEPTION 'Invalid grouping dimension: %', p_group_by_1.
- Build the same JOIN and WHERE clauses as RPC A (without OFFSET/LIMIT).
- Use `EXECUTE format(...)` to dynamically construct `SELECT %I, AVG(eff_price), MIN(eff_price), MAX(eff_price), AVG(CASE WHEN promo_price > 0 THEN promo_price END), MIN(CASE WHEN promo_price > 0 THEN promo_price END), MAX(CASE WHEN promo_price > 0 THEN promo_price END) FROM (...subquery...) t GROUP BY %I [, %I]`, where `%I` tokens receive the validated dimension name strings.
- Return `json_agg(row_to_json(g))`.

Run `python src/load_supabase.py` to provision. Verify: `SELECT proname FROM pg_proc WHERE proname IN ('get_landing_page_rows','get_landing_page_grouped');` returns 2 rows.

Grant EXECUTE to the `anon` role for both functions.

#### Done criteria
- `src/load_supabase.py` contains both DDL blocks.
- Both functions callable via `supabase.rpc(...)` without error.
- `get_landing_page_grouped` raises an exception when called with an invalid `p_group_by_1` value (e.g., `'injected; DROP TABLE'`).

#### Dependencies
Task 2.

#### Risk notes
The dynamic SQL whitelist check is security-critical. Use `IF NOT (p_group_by_1 = ANY(ARRAY[...])) THEN RAISE EXCEPTION ...`. The `EXECUTE format()` must use `%I` (not `%s`) for all identifier interpolations.
