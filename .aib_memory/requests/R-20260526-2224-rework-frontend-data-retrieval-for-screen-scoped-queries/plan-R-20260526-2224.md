## Goal

Restore the public landing page by replacing the current eager frontend data-fetch pattern with a screen-scoped retrieval model. The implementation must ensure that the landing page requests only the data needed for the active visible mode and that each frontend-visible Supabase query stays below the requested one-second budget.

## Constraints

- The implementation must preserve the current client-only React architecture documented in .aib_memory/context.md; no custom backend API layer may be introduced.

- The implementation must keep Supabase browser access limited to publishable-key-safe reads and must preserve the existing environment-variable contract in react-app/src/lib/supabase.js.

- The change must stay focused on the landing-page query path in react-app/src/App.jsx, react-app/src/components/LandingPage.jsx, react-app/src/lib/dataService.js, and src/load_supabase.py, plus directly related tests and documentation.

- The working interpretation of "current screen" is the active landing-page mode currently visible to the user: flat table or grouped view, plus only the selector datasets required for the active interaction.

- The working interpretation of the latency requirement is that each frontend-triggered Supabase request introduced or retained by this change must complete in under one second on the target dataset used by the maintainer when validating the fix.

## Success criteria

- The landing page in react-app/src/App.jsx and react-app/src/components/LandingPage.jsx renders successfully after the refactor instead of failing behind broad startup data loads.

- The active landing-page flow no longer issues unnecessary startup or refresh queries for data that is not needed by the currently visible screen state.

- The shared data layer in react-app/src/lib/dataService.js exposes only the query paths required by the active landing-page UX, and any removed count or eager-option paths are either deleted or no longer used by the screen.

- The Supabase SQL contract in src/load_supabase.py serves the narrowed landing-page request shapes with indexes or projection access paths that support the sub-second target.

- Frontend tests covering the changed behavior pass via `cd react-app && npm test`, and the production bundle still builds via `cd react-app && npm run build`.

- Documentation is updated in .aib_memory/context.md and any user-facing docs affected by the new query behavior.

## Plan

### Task 1: Rescope landing-page fetch orchestration

#### Intent

Move the landing-page UI off the current eager startup and refresh fan-out so only visible-mode data is requested at each interaction point.

#### Outputs

Updated react-app/src/App.jsx and react-app/src/components/LandingPage.jsx.

#### Procedure

Inspect react-app/src/App.jsx and remove any startup dependency on dimension data that the first visible landing-page render does not require.

Refactor react-app/src/components/LandingPage.jsx so the initial load requests only the flat rows or grouped rows needed for the visible mode, and so filter changes no longer force-refresh every option dataset unless the active interaction needs them.

Update react-app/src/components/LandingPage.jsx so selector data is loaded lazily or incrementally according to the visible control state, while preserving current error and loading-state behavior.

#### Done criteria

The landing page can render without waiting on the full legacy bootstrap payload, and a code review of react-app/src/components/LandingPage.jsx shows that each interaction triggers only the data fetches needed for the active visible state.

#### Dependencies

Task 2, Task 3.

#### Risk notes

LandingPage.test.jsx currently encodes eager option fetching on mount, so this refactor will require coordinated test updates to avoid false regressions.
