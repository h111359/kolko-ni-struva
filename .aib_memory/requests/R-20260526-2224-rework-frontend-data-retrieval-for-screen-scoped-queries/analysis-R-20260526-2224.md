## Input Interpretation

The developer requests restoration of the public React landing page by replacing the current broad database-fetch strategy with a screen-scoped retrieval model. The intended outcome is that the browser requests only the data required by the active landing-page view and active interaction mode, while all frontend-visible database calls stay below a one-second latency budget on the target Supabase dataset. This implies a change in both the frontend orchestration boundary and, where necessary, the Supabase query contracts that currently serve the landing page.

## Research Results

### Workspace pattern-scan findings

- react-app/src/App.jsx blocks first render on fetchDimensions(), so the landing page cannot render until startup dimension queries complete.

- react-app/src/lib/dataService.js makes seven startup reads inside fetchDimensions(): dim_date, dim_settlement, dim_category, dim_store, dim_company, dim_file, and get_available_dates(). That startup path is broader than the visible landing-page state requires on first paint.

- react-app/src/components/LandingPage.jsx triggers five option RPCs plus one flat-row RPC on initial mount, and repeats the same five-option fan-out on filter refresh even though only one screen is visible and only one data mode is active at a time.

- react-app/src/components/LandingPage.jsx switches between flat rows and grouped rows, but it still refreshes option lists on every triggerRefresh() call regardless of whether those option lists are currently stale, visible, or needed for the active interaction.

- react-app/src/lib/dataService.js keeps a fetchLandingPageCount() helper and its test coverage even though the active landing-page paginator no longer depends on exact total counts. That dead path increases maintenance surface around a query the active screen does not need.

- react-app/src/components/LandingPage.test.jsx codifies the current eager behavior by asserting that all five option RPCs run on mount. Those tests will need to change if the app is refactored toward lazy, current-screen-only reads.

- src/load_supabase.py already provisions a landing_page_row_projection table plus projection indexes and RPCs, which means the repository already favors database pushdown and read-optimized projections. The remaining inefficiency is not the absence of a backend contract; it is that the frontend still requests too many separate datasets per interaction and several option RPCs still scan fact_prices_lookback with joins rather than the projection.

### Industry knowledge

- React application guidance from the React ecosystem and route-loader patterns consistently recommends fetching critical-path data at the screen boundary and deferring secondary data until it is needed. Applicability: high. The current landing page issues startup and refresh requests for data that is not strictly required for the first visible render, so a lazy screen-boundary fetch strategy is aligned with established SPA practice.

- Supabase and PostgREST performance guidance consistently favors narrow column selection, filtered server-side RPCs, and pagination over broad client-side hydration of reference data. Applicability: high. The current landing page already uses server-side row and grouped RPCs, but it still fans out into multiple option RPCs and startup dimension reads that exceed the minimum data needed for the visible state.

- PostgreSQL performance practice favors stable query shapes that match composite indexes and avoids repeated DISTINCT-plus-join option lookups when a smaller projection or cached bootstrap payload can satisfy the same screen. Applicability: high. The repository already carries targeted landing-page indexes, so the next improvement is to reduce redundant option-query traffic and ensure remaining option queries read from the cheapest indexed surface.

- Frontend performance guidance from RAIL-style interaction budgets treats sub-second response time as the upper bound for uninterrupted user flow, with faster responses preferred for direct manipulation. Applicability: high. The developer supplied an explicit sub-second query target, so both frontend orchestration and SQL contracts should be evaluated against that latency budget rather than against correctness alone.

## Decision Register

### Decision Points

#### Decision Point: Data retrieval boundary for the landing page

- Applies to: the first render path in react-app/src/App.jsx and the active interaction refresh path in react-app/src/components/LandingPage.jsx and react-app/src/lib/dataService.js.

- Why alternatives exist: the app can either continue preloading broad shared datasets and every cross-filter option set, or it can fetch only the data required by the currently visible landing-page mode and requested page.

- Chosen alternative: use screen-scoped, mode-specific retrieval so the landing page requests only the current visible result set and only the selector data that is necessary for the active interaction.

- Key trade-offs: this reduces query count and latency on the critical path, but it requires tighter coordination between frontend state transitions and the Supabase RPC surface.

- Expected codebase impact: react-app/src/App.jsx, react-app/src/components/LandingPage.jsx, react-app/src/lib/dataService.js, src/load_supabase.py, and related tests will change together.

- Tag: resolved-by-user

- Rationale/Resolution: the developer explicitly requested, "Make the queries to retrieve just the data needed for the current screen. Each query should run less than a second" in .aib_memory/input.md.

#### Decision Point: Latency target for the reworked query path

- Applies to: every changed Supabase call that serves the landing page.

- Why alternatives exist: the implementation could optimize only for correctness, or it could treat response time as an explicit acceptance requirement for the new data-access design.

- Chosen alternative: enforce a sub-second target for each frontend-visible landing-page query on the target dataset.

- Key trade-offs: this sets a measurable acceptance bar, but it may require both frontend request reduction and backend SQL/index changes instead of a frontend-only refactor.

- Expected codebase impact: validation steps, query logging, SQL/index design, and request orchestration must all be checked against the same latency budget.
