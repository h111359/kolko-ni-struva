## Input

### Goal

Replace the complete Supabase-based architecture with a Microsoft Azure-based architecture centered on Azure SQL Database while preserving all existing business functionality and user-visible behavior unless explicitly required by the new architecture.

This is a complete migration. Supabase shall be removed entirely from the project.

### Architecture

Implement the following target architecture:

```
Local ETL
    ↓
load_azure_sql.py
    ↓
Azure SQL Database
    ↓
Azure Functions (Python)
    ↓
REST API (OpenAPI-first)
    ↓
React application
```

The local ETL output remains the authoritative data source.

Azure SQL is the serving database.

The React application shall no longer communicate directly with a database and shall instead communicate exclusively with the REST API exposed by Azure Functions.

### Database

Replace the existing PostgreSQL/Supabase implementation with Azure SQL Database.

Use the existing Azure SQL database:

* Server: sql-server-kolko-ni-struva
* Database: sql-db-kolko-ni-struva

Use schema:

* dbo

The implementation shall target Azure SQL while avoiding unnecessary dependencies on compatibility-level-170-specific features.

Rewrite all PostgreSQL-specific SQL into idiomatic T-SQL.

Replace PostgreSQL concepts including, but not limited to:

* data types
* indexes
* functions
* sequences
* UPSERT logic
* RETURNING
* ON CONFLICT
* DISTINCT ON
* ILIKE
* PostgreSQL-specific syntax
* Supabase-specific constructs

with their SQL Server equivalents.

### ETL

Replace `load_supabase.py` with `load_azure_sql.py`.

Preserve the existing ETL workflow:

```
extract.py
    ↓
transform.py
    ↓
load_azure_sql.py
```

The Azure loading stage shall remain independent from transformation.

Use:

* pyodbc
* ODBC Driver 18 for SQL Server
* fast_executemany

Authentication for ETL shall use SQL authentication with the existing SQL login.

Dimensions shall be synchronized incrementally.

Fact tables shall be bulk reloaded.

The implementation shall optimize loading performance while minimizing transaction log growth.

### Data Retention

The local star schema remains the authoritative data store.

Azure SQL shall contain a configurable rolling window of historical fact data.

Default retention shall be three days.

The retention period shall be configurable rather than hardcoded.

### Database Logic

Move the existing PostgreSQL RPC functionality into SQL Server stored procedures.

Business logic shall reside in SQL Server wherever practical.

Implement a single parameterized search stored procedure supporting optional filters, grouping, sorting and paging.

Azure Functions shall call stored procedures rather than constructing SQL dynamically.

### REST API

Create a new REST API implemented using Azure Functions (Python).

The API shall be designed OpenAPI-first.

Generate a complete OpenAPI specification.

The REST API shall be designed specifically for Azure rather than attempting to emulate Supabase APIs.

Implement clean REST endpoints appropriate for the application.

The React application shall be updated accordingly.

### Security

Azure Functions shall authenticate to Azure SQL using Managed Identity.

The REST API shall be anonymous read-only.

No Azure API Management shall be introduced.

Protect the API through:

* strict request validation
* bounded page sizes
* parameter validation
* parameterized stored procedure execution
* robust error handling

### React Application

Remove all Supabase dependencies.

Remove:

* @supabase/supabase-js
* Supabase client code
* RPC calls
* PostgREST-specific logic

Implement a new REST client communicating with Azure Functions.

Update the React application to consume the new API.

### Infrastructure

Generate complete Azure infrastructure as code using Bicep.

Include all required Azure resources including:

* Azure Functions
* hosting plan
* storage account
* Application Insights
* managed identity
* configuration
* role assignments

Generate deployment scripts for:

* Bash
* PowerShell

Generate complete deployment instructions describing:

* prerequisites
* deployment
* configuration
* SQL deployment
* verification
* rollback

### Documentation

Update all project documentation to reflect the Azure architecture.

Remove obsolete Supabase documentation.

Create an Architecture Decision Record describing the migration from Supabase to Azure.

The ADR shall map every major Supabase capability to its Azure replacement.

### Migration

Perform a complete migration.

After implementation there shall be no remaining production dependency on:

* Supabase
* PostgreSQL
* PostgREST
* Supabase RPC
* Supabase JavaScript client

unless retained exclusively inside archived documentation or migration notes.

### Analysis

Before implementation:

* analyze the existing implementation in detail
* identify every Supabase dependency
* identify every PostgreSQL-specific SQL construct
* identify every React component affected
* identify every ETL component affected
* produce a comprehensive implementation plan
* estimate migration complexity
* identify migration risks
* identify all files that require modification
* propose any improvements that naturally fit the Azure architecture while preserving existing functionality.
