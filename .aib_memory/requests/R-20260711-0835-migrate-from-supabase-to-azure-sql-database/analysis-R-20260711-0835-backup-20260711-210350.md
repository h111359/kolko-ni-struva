# Analysis Document

**Request ID:** R-20260711-0835  
**Request Title:** Migrate from Supabase to Azure SQL Database

## Overview

### Background

The current product architecture uses Supabase as the cloud database and API layer, with PostgreSQL as the underlying database engine and PostgREST providing REST API access. The React application communicates directly with Supabase using the Supabase JavaScript client library, executing RPC calls against PostgreSQL functions for data retrieval and filtering. This architectural pattern was chosen for rapid development and minimal backend infrastructure requirements.

The migration to Azure SQL Database is driven by a strategic decision to consolidate cloud infrastructure on Microsoft Azure rather than maintaining multiple cloud providers. This migration requires replacing the entire Supabase stack with Azure-native equivalents while preserving all existing business functionality and user-visible behavior.

### Scope

This migration encompasses:

- Complete replacement of PostgreSQL database with Azure SQL Database (server: sql-server-kolko-ni-struva, database: sql-db-kolko-ni-struva)
- Rewriting src/load_supabase.py as src/load_azure_sql.py using pyodbc and ODBC Driver 18
- Converting all PostgreSQL-specific SQL constructs to T-SQL equivalents (data types, indexes, functions, sequences, UPSERT, RETURNING, ON CONFLICT, DISTINCT ON, ILIKE, and Supabase-specific constructs)
- Migrating 11 PostgreSQL RPC functions to SQL Server stored procedures
- Creating new REST API using Azure Functions (Python) with OpenAPI-first design
- Implementing Managed Identity authentication from Azure Functions to Azure SQL
- Removing all Supabase dependencies from React application (@supabase/supabase-js, client code, RPC calls, PostgREST logic)
- Implementing new REST client in React to consume Azure Functions API
- Generating Azure infrastructure as code using Bicep (Azure Functions, hosting plan, storage account, Application Insights, managed identity, role assignments)
- Generating deployment scripts for Bash and PowerShell
- Creating comprehensive deployment instructions
- Updating all project documentation to reflect Azure architecture
- Creating Architecture Decision Record mapping Supabase capabilities to Azure replacements
- Updating menu.py to support Azure SQL deployment instead of Supabase sync
- Modifying tests to work with new Azure architecture

### Out of Scope

This migration explicitly excludes:

- Changes to extract.py or transform.py ETL stages (these remain unchanged)
- Modifications to local star schema structure or file formats
- Changes to data/nomenclatures/ reference files
- Alterations to quality reporting logic
- Introduction of Azure API Management
- Implementation of authentication/authorization beyond anonymous read-only REST API
- Changes to Netlify deployment process for React application
- Migration of historical data beyond configured retention window
- Support for multiple Azure environments (dev/staging/prod)

### AI Interpretation

This request specifies a comprehensive platform migration from a Supabase-PostgreSQL architecture to a Microsoft Azure-SQL Server architecture. The migration scope is complete and irreversible: all Supabase and PostgreSQL dependencies must be eliminated from the production codebase.

The target architecture introduces a three-tier separation: (1) local Python ETL producing CSV outputs, (2) Azure SQL Database serving as the cloud data warehouse, and (3) Azure Functions providing a REST API layer between the database and the React frontend. This represents a significant architectural shift from the current direct-database-access pattern where the React application uses Supabase's client library to execute RPC calls directly against PostgreSQL functions.

The migration must preserve functional equivalence: all existing business capabilities, user-visible behaviors, and data operations must continue to work identically from the end user's perspective. The only acceptable behavioral changes are those inherently required by the new architecture (such as REST API URLs replacing Supabase RPC endpoints).

Key technical constraints include: (1) use of existing Azure SQL infrastructure rather than provisioning new resources, (2) SQL authentication for ETL rather than Windows authentication, (3) Managed Identity authentication for Azure Functions, (4) anonymous read-only REST API without Azure API Management, (5) OpenAPI-first API design rather than Supabase API emulation, (6) configurable rolling retention window with three-day default, (7) avoidance of SQL Server 2022-specific features to maintain compatibility with existing database compatibility level.

The migration must be implemented atomically: there is no requirement to support a dual-stack transition period where both Supabase and Azure coexist. The local ETL and React application should continue to function throughout the migration, but once the migration is complete, all Supabase dependencies must be removed.

### Industry Knowledge

**PostgreSQL to T-SQL Migration Patterns:**

1. PostgreSQL SERIAL becomes SQL Server IDENTITY: PostgreSQL SERIAL columns must be converted to INT IDENTITY(1,1) or BIGINT IDENTITY(1,1). The RETURNING clause used after INSERT must be replaced with SCOPE_IDENTITY() or OUTPUT clause.

2. ON CONFLICT becomes MERGE: PostgreSQL INSERT...ON CONFLICT DO UPDATE must be rewritten as MERGE statements in T-SQL. This is a one-to-one semantic mapping but requires restructuring the SQL statement.

3. DISTINCT ON becomes ROW_NUMBER: PostgreSQL DISTINCT ON must be rewritten using ROW_NUMBER() OVER(PARTITION BY ... ORDER BY ...) in a CTE or subquery. This is a common transformation pattern in PostgreSQL-to-SQL Server migrations.

4. ILIKE becomes COLLATE: PostgreSQL case-insensitive ILIKE operator must be replaced with LIKE combined with a case-insensitive collation (e.g., COLLATE Latin1_General_CI_AI) or converted to LOWER() functions on both sides of comparison.

5. Array aggregation becomes FOR XML or STRING_AGG: PostgreSQL array_agg() and array operations must be converted to STRING_AGG() (SQL Server 2017+) or FOR XML PATH workarounds for older versions.

6. Function return types: PostgreSQL functions returning TABLE must become table-valued functions using RETURNS TABLE in SQL Server. Multi-statement table-valued functions require explicit RETURN statements.

**Azure Functions Best Practices for Database APIs:**

1. Use Managed Identity for authentication: Azure Functions should authenticate to Azure SQL using Managed Identity rather than connection strings. This eliminates secret management and improves security posture.

2. Connection pooling: Azure Functions Python runtime does not automatically pool database connections. Implement application-level connection pooling using singleton pattern or dependency injection to avoid connection exhaustion.

3. Stored procedure pattern: Calling parameterized stored procedures is more secure and performant than dynamic SQL construction. All business logic should reside in stored procedures, with Azure Functions acting as a thin REST-to-database adapter.

4. OpenAPI-first design: Generate OpenAPI specification first, then implement Azure Functions to match. This ensures API consistency and enables automatic client generation.

5. Error handling: Distinguish between client errors (400 Bad Request) and server errors (500 Internal Server Error). Validate all inputs before calling database. Return structured error responses with correlation IDs for traceability.

**Azure SQL Deployment Patterns:**

1. Use SQLCMD mode for deployment scripts: Generate .sql files with SQLCMD variables for environment-specific values (server name, database name). This enables parameterized deployment across environments.

2. Idempotent DDL: Use DROP IF EXISTS or CREATE OR ALTER patterns to make deployment scripts rerunnable. Essential for iterative development and rollback scenarios.

3. Transaction log management: Bulk operations should use batch processing to avoid transaction log growth. Set RECOVERY model to SIMPLE for data warehouse workloads with external backup.

### Technical Notes

**SQL Server Connection Pooling:** The pyodbc library does not implement connection pooling by default. The load_azure_sql.py implementation should use a singleton pattern or context manager to maintain a single connection throughout the sync operation. For Azure Functions, implement application-level connection pooling using a global connection pool initialized once per Function App instance.

**Managed Identity Authentication:** Azure Functions will use Managed Identity to authenticate to Azure SQL. This requires: (1) enabling System Managed Identity on the Function App, (2) creating a SQL user mapped to the Managed Identity using CREATE USER [function-app-name] FROM EXTERNAL PROVIDER, (3) granting appropriate roles (db_datareader, db_datawriter), (4) using authentication=ActiveDirectoryMsi in the pyodbc connection string.

**MERGE Statement Performance:** SQL Server MERGE statements can be slower than separate INSERT/UPDATE logic for small datasets due to full table scans. For dimension tables with thousands of rows, MERGE performance is acceptable. For larger dimensions (>100K rows), consider implementing separate UPDATE/INSERT logic with indexed lookups.

**Transaction Log Growth:** Bulk INSERT operations on fact_prices_lookback can cause significant transaction log growth. Implement batched inserts with COMMIT after each batch (e.g., 10,000 rows per batch) to allow log truncation. Monitor log space usage and consider setting database to SIMPLE recovery model if point-in-time restore is not required.

**Error Handling Strategy:** Azure Functions should distinguish between retriable errors (database timeout, connection failure) and non-retriable errors (invalid parameters, missing data). Return 503 Service Unavailable for retriable errors and 400 Bad Request for client errors. Implement retry logic with exponential backoff in the React application for 503 responses.

**CORS Configuration:** Azure Functions requires explicit CORS configuration to allow browser requests from Netlify-hosted React application. Add Netlify domain to allowed origins in Function App CORS settings. For local development, allow localhost:5173 (Vite default port).
