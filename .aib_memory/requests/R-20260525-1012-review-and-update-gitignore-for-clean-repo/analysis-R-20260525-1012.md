# Analysis: R-20260525-1012

## Overview

**Request ID:** R-20260525-1012  
**Request title:** Review and update .gitignore for clean repo

### Background

The developer wants the repository to be publishable to GitHub in a clean state: only application source code and essential configuration committed; data, secrets, tooling artifacts, and generated outputs excluded. The motivation is twofold — (a) the repository currently contains sensitive credentials in tracked files (a Netlify API token and a Supabase service_role key with full database access and a PostgreSQL password), and (b) several non-code artefacts (node_modules, lab data, Netlify CLI local state, test output) are tracked unnecessarily, inflating repository size and creating confusion about what is required for fresh installation.

### Scope

- Review the existing `.gitignore` and identify all gaps.
- Identify all tracked files that should not be tracked (non-application-code artefacts and credential files).
- Determine required additions to `.gitignore` to prevent future accidental commits of those files.
- Assess whether the application can be fully restored from the cleaned repository on a new machine, and identify any missing setup artefacts.
- Rewrite git history to remove both credential files from all historical commits; rotate all three exposed credentials (Q001 — resolved).
- Exclude `lab/` from tracking (Q002 — resolved).
- Produce `config.ini.example` and exclude `config.ini` from tracking, following `.env` / `.env.example` pattern (Q003 — resolved).
- Add `!data/nomenclatures/` exception to `.gitignore` so EKATTE reference files are committed (Q004 — resolved).

### Out of scope

- Changes to application source code (`src/`, `react-app/src/`, `tests/`, `menu.py`, etc.).
- Changes to ETL pipeline logic or React app functionality.
- Changes to CI/CD configuration (`react-app/netlify.toml`).
- Automated scheduling or cloud infrastructure setup.
- Backfilling or migrating historical git objects beyond the two credential files being purged.

### Industry Knowledge

**Finding 1 — GitHub secret scanning and credential rotation (GitHub / GitHub Docs, 2023).**  
GitHub's secret scanning feature automatically detects committed credentials (API tokens, JWTs, database URLs) and can notify the secret provider (in this case Netlify and Supabase). For Supabase, the `service_role` key carries full bypass of row-level security; once exposed, the key must be treated as compromised regardless of whether the repo is private. Industry practice is to immediately rotate any exposed credential, not to rely on the repo remaining private. Applicability: CRITICAL — the Supabase service_role key must be rotated even if the repo is never made public.

**Finding 2 — Removing sensitive data from git history (`git filter-repo` / BFG Repo-Cleaner).**  
GitHub's official documentation recommends `git filter-repo` (a fast Python tool) or BFG Repo-Cleaner to rewrite git history and remove specific files or content strings from all commits. Both tools require a force-push to all remotes and all collaborators must re-clone or rebase. This is a destructive but effective remediation. The alternative — leaving secrets in history but revoking credentials — is acceptable only for private repos when force-push is not feasible, but is insufficient for repos that may become public. Applicability: HIGH — a decision is needed from the developer on which approach to take.

**Finding 3 — `node_modules` in git repositories (npm / community best practice).**  
The npm documentation and the broader JavaScript community universally recommend against committing `node_modules/` to version control. `package.json` and `package-lock.json` (both tracked) are sufficient for deterministic restoration via `npm ci` (or `npm install`). Committing `node_modules/` inflates repository size, creates merge conflicts, and is a known source of platform-specific binary corruption. Applicability: DIRECT — `react-app/node_modules/` must be removed from tracking and added to `.gitignore`.

**Finding 4 — Netlify CLI state directories.**  
The `.netlify/` directory is generated by the Netlify CLI and records local state (site linkage, deploy URLs, feature flags). It is machine-specific and is listed in Netlify's own `.gitignore` template as an exclusion. The `netlify.toml` in `react-app/` is the correct deployment configuration to commit; `.netlify/netlify.toml` is a local override and should not be committed. Applicability: DIRECT — `.netlify/` should be excluded.

**Finding 5 — Static reference data and the "installability" principle (12-factor app methodology).**  
The 12-factor app methodology distinguishes between configuration (environment-specific, from env vars) and code (static, committed). Static public reference data (e.g., government-published EKATTE tables) occupies a grey zone: it is not generated, not secret, and does not change frequently. For small datasets, committing them as part of the repository with a clear directory structure is the most reliable approach for ensuring fresh-install operability. The alternative — downloading them as part of a setup script — requires internet access and external URL stability. Applicability: MEDIUM — tracking `data/nomenclatures/` directly, with a clear exception in `.gitignore`, is the most reliable path for fresh-install operability.

### AI Agent Critique

- **`netlify token.txt` file name contains spaces**, making it awkward to handle in scripts and shells. Even after excluding it from the repo, this file should be renamed or its contents should be migrated to `.env` or a secrets manager.
- **`curl https ekootljybgoenduwprbw.txt` file name is a literal curl command** used as scratch notes — a debugging artifact that was accidentally staged and committed. This pattern (committing scratch/debug files) indicates the developer does not have a disciplined pre-commit review workflow; a pre-commit hook checking for credential patterns (e.g., `pre-commit` with `detect-secrets` or `gitleaks`) would prevent recurrence.
- **The Supabase service_role key in the curl file has an expiry of 2091** (decoded from the JWT), meaning it will be valid for approximately 65 years. This maximizes the window of exposure if the key is not rotated.
- **`.netlify/netlify.toml` hardcodes the developer's home directory** (`/home/hromar/Desktop/projects/kolko-ni-struva`), which would break Netlify CLI usage for any collaborator who clones the repo and has the `.netlify/` directory present.
- **`react-app/node_modules/` contains 3,306 tracked files.** Removing these from tracking requires `git rm -r --cached react-app/node_modules/` followed by a commit, which will produce a very large commit (3,306 deletions). If the repo is already pushed to a remote, this will be a large push. This is a one-time cost but is unavoidable.
- **The `data/` gitignore exclusion uses both the parent directory and explicit subdirectory entries** (`data/raw/`, `data/interim/`, `data/processed/`). Git ignores subdirectory entries when the parent is already excluded; these explicit subdirectory lines are dead rules.
- **`lab/` is tracked but `data/` is excluded**, creating an inconsistency: the lab folder appears to serve the same exploratory/raw-data purpose as `data/`, yet one is tracked and the other is not. The intent of `lab/` needs clarification.
- **`react-app/test_output.txt` should not be committed.** Test output files are transient and machine-specific; they clutter the repository and can cause spurious diffs.
- **`README.md` does not document the fresh-install procedure** for a machine that clones the repo for the first time (e.g., which steps to take: pip install, configure .env, obtain nomenclature files, npm install in react-app/, etc.). A gap here means the "restorable from repo" goal is partially incomplete at the documentation level.
- **`config.ini` state dates** (`last_downloaded_date`, `last_processed_date`) are machine-written. On a fresh clone these dates would cause the ETL to skip all data older than 2026-05-24 and treat that as already processed. For a developer setting up on a new machine from scratch, this means the first ETL run starts from the current date rather than re-processing from the beginning, which may or may not be desired.
- **The `.gitignore` `kolko-ni-struva/` entry** suggests a historical period when this repo was nested inside a parent directory named `kolko-ni-struva`. This entry is now misleading and functionally a no-op from the repo root.

#### Decision Point: Handling credentials already in git history

The credentials in `netlify token.txt` and `curl https ekootljybgoenduwprbw.txt` are in committed git history. Simply adding `.gitignore` entries does not remove them from existing commits.

- **Tag:** `resolved-by-user`
- **Chosen alternative:** Option A — History rewrite + credential rotation. Use `git filter-repo --path "netlify token.txt" --path "curl https ekootljybgoenduwprbw.txt" --invert-paths` to remove both files from every historical commit, force-push to all remotes, and simultaneously revoke/rotate all three exposed credentials (Netlify token, Supabase service_role key, PostgreSQL password).
- **Resolution:** Q001 — developer selected Option A.
