## Goal

Replace the batched-INSERT full refresh of `fact_prices_lookback` with a COPY-based full refresh using `psycopg2`'s `copy_expert()`, and replace the physical `landing_page_row_projection` table and its recurring rebuild cycle with a regular PostgreSQL view, while keeping `src/load_supabase.py` as a single self-provisioning sync command. The intent is to reduce WAL generation during the Supabase sync run to a level that does not trigger free-tier read-only mode.

## Constraints

- Python 3.9 or newer; psycopg2 direct PostgreSQL connection via `DATABASE_URL`.
- `src/load_supabase.py` must remain a single self-provisioning command that provisions schema, syncs data, and runs pruning in one invocation.
- `COPY FROM STDIN` must handle empty-string-to-NULL coercion via the `NULL ''` COPY option, maintaining behavioral equivalence with the current `_coerce()` path.
- The view `landing_page_row_projection` must expose the same column names and semantics as the current physical table so that all RPC functions consuming it (`get_landing_page_rows`, all five `get_lp_options_*`) require no SQL-level changes.
- `GRANT SELECT ON landing_page_row_projection TO anon` must remain in `_CREATE_RPC_FUNCTIONS` to maintain browser-accessible read permissions.
- No changes to `src/extract.py`, `src/transform.py`, `menu.py`, or any React frontend file.
- Two pre-existing test failures (`test_index_ddl_contains_lookback_composite_date_store_index` and `test_index_ddl_contains_report_slice_index`) must be removed because the indexes they assert do not exist in the current codebase.

## Success criteria

- `python src/load_supabase.py` completes without error on a populated local dataset and without raising a psycopg2 database error.
- `python -m pytest tests/test_load_supabase.py` passes with zero failures and zero errors.
- After sync, `landing_page_row_projection` exists as a VIEW (not a TABLE) in the Supabase PostgreSQL schema.
- `fact_prices_lookback` is populated via the `copy_expert` COPY path; `execute_batch_rows` is no longer called by `insert_lookback()`.
- `refresh_landing_page_projection` does not appear as a callable in `main()` and does not exist as a function in `src/load_supabase.py`.
- `README.md` describes the COPY-based fact load and the view-based projection.
- `.aib_memory/context.md` correctly describes `landing_page_row_projection` as a PostgreSQL VIEW.

## Plan

### Task 1: Replace physical projection table with a view in src/load_supabase.py

#### Intent
Replace all projection-table-specific constructs in `src/load_supabase.py` with a view definition and remove the recurring TRUNCATE + INSERT projection rebuild cycle.

#### Outputs
`src/load_supabase.py` — modified `_MIGRATION_DDL`, `_CREATE_DDL`, `_CREATE_INDEXES`; removed `_REFRESH_LANDING_PAGE_PROJECTION_SQL`, `_REFRESH_LANDING_PAGE_PROJECTION_BATCH_SQL`, `PROJECTION_REFRESH_BATCH_SIZE`; removed `refresh_landing_page_projection()` function; removed step 7 from `main()`.

#### Procedure

In `src/load_supabase.py`, add the line `DROP TABLE IF EXISTS landing_page_row_projection CASCADE;` to the `_MIGRATION_DDL` string. This ensures existing Supabase databases drop the physical projection table before the view creation attempt in `_CREATE_DDL`.

In `src/load_supabase.py`, replace the `CREATE TABLE IF NOT EXISTS landing_page_row_projection (…)` block at the end of `_CREATE_DDL` with the following view definition, using the same SELECT body from `_REFRESH_LANDING_PAGE_PROJECTION_SQL`:

```sql
CREATE OR REPLACE VIEW landing_page_row_projection AS
SELECT
    f.date_key,
    dstore.settlement_key,
    f.category_key,
    dstore.company_key,
    f.store_key,
    f.file_key,
    f.product_key,
    df.file_name,
    dp.product_name,
    dc.category_name,
    dst.settlement_name,
    dstore.store_name,
    dcomp.company_name,
    f.retail_price,
    f.promo_price,
    COALESCE(
        CASE
            WHEN f.promo_price IS NOT NULL AND f.promo_price > 0
                THEN LEAST(f.retail_price, f.promo_price)
            ELSE f.retail_price
        END,
        f.retail_price
    ) AS price
FROM fact_prices_lookback f
JOIN dim_product dp ON dp.product_key = f.product_key
JOIN dim_category dc ON dc.category_key = f.category_key
JOIN dim_store dstore ON dstore.store_key = f.store_key
JOIN dim_company dcomp ON dcomp.company_key = dstore.company_key
JOIN dim_settlement dst ON dst.settlement_key = dstore.settlement_key
LEFT JOIN dim_file df ON df.file_key = f.file_key;
```

In `src/load_supabase.py`, remove the three index creation statements for the projection table from `_CREATE_INDEXES`: the blocks defining `idx_lp_row_projection_page`, `idx_lp_row_projection_filter_keys`, and `idx_lp_row_projection_product_name_trgm`.

In `src/load_supabase.py`, delete the module-level string constants `_REFRESH_LANDING_PAGE_PROJECTION_SQL`, `_REFRESH_LANDING_PAGE_PROJECTION_BATCH_SQL`, and the integer constant `PROJECTION_REFRESH_BATCH_SIZE = 250`.

In `src/load_supabase.py`, delete the `refresh_landing_page_projection()` function entirely.

In `src/load_supabase.py` `main()`, remove step 7: delete the `print("Refreshing landing-page projection …")` line and the `refresh_landing_page_projection(conn)` call.

In `src/load_supabase.py` `create_tables()` docstring, correct the execution-order list to match the actual code order: (1) `_MIGRATION_DDL`, (2) `_CREATE_DDL`, (3) `_ENSURE_NULLABLE_DDL`, (4) `_CREATE_RPC_FUNCTIONS`, (5) `_CREATE_INDEXES`.

In `src/load_supabase.py` module docstring (first block), add a reference to request R-20260612-0710.

#### Done criteria
- `_CREATE_DDL` contains `CREATE OR REPLACE VIEW landing_page_row_projection` and does not contain `CREATE TABLE IF NOT EXISTS landing_page_row_projection`.
- `_MIGRATION_DDL` contains `DROP TABLE IF EXISTS landing_page_row_projection CASCADE`.
- `_CREATE_INDEXES` does not contain `idx_lp_row_projection_page`, `idx_lp_row_projection_filter_keys`, or `idx_lp_row_projection_product_name_trgm`.
- `refresh_landing_page_projection` does not appear in `main()` as a callable.
- `_REFRESH_LANDING_PAGE_PROJECTION_SQL`, `_REFRESH_LANDING_PAGE_PROJECTION_BATCH_SQL`, `PROJECTION_REFRESH_BATCH_SIZE` do not appear in the file.
- Module syntax check passes: `python -c "import sys; sys.path.insert(0,'src'); import load_supabase"` succeeds without error.

#### Dependencies
None.

#### Risk notes
`CREATE OR REPLACE VIEW` cannot replace an existing TABLE in PostgreSQL — it will raise an error if a table with that name already exists. The `DROP TABLE IF EXISTS … CASCADE` in `_MIGRATION_DDL` is the migration guard. The existing code always runs `_MIGRATION_DDL` before `_CREATE_DDL` in `create_tables()`, so this is low risk under normal execution but must be verified in the test that checks DDL statement ordering.

### Task 2: Replace batched INSERT with COPY in insert_lookback()

#### Intent
Replace the `TRUNCATE + execute_batch_rows()` fact-load path in `insert_lookback()` with `TRUNCATE + cur.copy_expert()` using a server-side COPY command, reducing per-row write overhead and WAL generation for the largest recurring write operation.

#### Outputs
`src/load_supabase.py` — modified `insert_lookback()` function body.

#### Procedure

In `src/load_supabase.py` `insert_lookback()`, replace the section that builds the `insert_sql` variable, accumulates `rows` with `csv.DictReader`, and calls `execute_batch_rows(cur, insert_sql, rows)` with the following COPY-based approach:

Keep the `TRUNCATE TABLE fact_prices_lookback` execute call unchanged inside the transaction block.

Replace the row-accumulation and batch-insert block with: when `csv_path.exists()`, open the file in text mode (`encoding="utf-8", newline=""`) and call `cur.copy_expert(copy_sql, fh)` where `copy_sql` is:
```
f"COPY fact_prices_lookback ({col_list}) FROM STDIN WITH (FORMAT CSV, HEADER, NULL '')"
```
and `col_list` is the comma-joined `columns` list already defined at the top of the function.

After `copy_expert`, capture `cur.rowcount` as `row_count` (fallback to 0 if `csv_path` does not exist, fallback to 0 if rowcount is -1).

Remove the `insert_sql` variable and the `rows: List[tuple] = []` accumulation block.

Update the `print` statement and `return` to use `row_count` instead of `len(rows)`.

Remove the `csv` import usage within this function if `csv.DictReader` was the only CSV consumer (verify whether `upsert_dim()` still uses `csv.DictReader` — if yes, keep the `import csv` at the module level).

#### Done criteria
- `insert_lookback()` does not call `execute_batch_rows()` or build a `rows` list.
- `insert_lookback()` calls `cur.copy_expert()` with a SQL string containing `COPY fact_prices_lookback` and `FROM STDIN`.
- Module syntax check passes without error.

#### Dependencies
Task 1 (modifies the same file; apply in order to minimize conflicts).

#### Risk notes
`psycopg2`'s `copy_expert()` is available in all psycopg2 versions ≥ 2.0. The `requirements.txt` specifies `psycopg2-binary` without a version pin; confirm the installed version at implementation time. `cur.rowcount` after `COPY` returns -1 on some older psycopg2 versions if the count cannot be determined; the implementation must guard against -1 (treat as 0 or log as unknown) to avoid confusing output.
