### Background

The `dim_settlement` dimension table (`data/schema/dim_settlement.csv`) and its Supabase mirror contain two categories of data quality defects introduced by earlier pipeline runs:

1. **EKATTE leading-zero duplicates.** The `normalize_settlement_code()` function was introduced in R-20260508-0743 to canonicalize settlement codes before `dim_settlement` natural-key assignment, but it was not applied retroactively. As a result, 15 settlement groups each appear twice — once under a non-canonical EKATTE variant (e.g., `357`) and once under the canonical form (e.g., `00357`). Each variant received a distinct `settlement_key`, causing analytical splits. Thirty stores in `dim_store.csv` reference the retired (non-canonical) settlement keys.

2. **Duplicate `settlement_name` values and missing structural metadata.** After removing the 15 EKATTE-duplicate pairs, 5 genuine same-name pairs remain among distinct real-world settlements. The developer has additionally requested that all settlement names be reformatted to a fully-qualified "type name (oblast)" label (e.g., "гр. Велико Търново (обл. Велико Търново)") for all rows regardless of whether a duplicate existed, and that two new columns — `settlement_type` and `oblast` — be added to `dim_settlement` so the React app can sort dropdowns by base name, then type, then oblast. For Sofia raion entries the format is "Sofia, р-н {raion_name}" (confirmed by Q005). The combined result is that `settlement_name` is unique across all 256 post-fix rows, and each row carries explicit type and oblast metadata.

## Input Interpretation

The data engineer has identified two distinct data quality defects in the `dim_settlement` dimension that cause incorrect analytical grouping and ambiguous user-facing settlement labels. The Q004 answer has substantially expanded the scope of the name-correction work.

**Defect 1 — EKATTE leading-zero duplication.** Source data CSVs contain EKATTE settlement codes in non-canonical forms. When `normalize_settlement_code()` was introduced in R-20260508-0743, it began canonicalizing new incoming codes correctly but was not retroactively applied. The same real-world settlement was therefore inserted multiple times under different EKATTE formats, each receiving a distinct `settlement_key`. A total of 15 duplicate groups are present (271 current rows; 256 after removal), and 30 `dim_store` rows reference the "retired" (non-canonical) settlement keys. The developer requires a surgical fix that removes these duplicates while preserving surrogate key stability.

**Defect 2 — Non-unique `settlement_name` values.** After the EKATTE deduplication removes 15 rows, 5 genuine same-name pairs remain among distinct settlements: Бяла ×2 (two cities), Разград ×2, Елин Пелин ×2, Габрово ×2, and Средец ×2 (city + Sofia raion). The developer's primary goal stated as "the settlement_name to be unique" requires resolving all five pairs.

**Q004 scope expansion — Fully-qualified labels for all settlements and new schema columns.** Through Q004, the developer expanded the scope beyond duplicate disambiguation: ALL settlement labels (not just duplicates) should follow a "type name (oblast)" format, regardless of whether a duplicate existed. Additionally, `settlement_type` and `oblast` should be stored as separate columns in `dim_settlement` so that React app dropdown lists can be ordered by base name (alphabetically), then by type, then by oblast. This ensures settlements like "гр. Търговище (обл. Търговище)" and "с. Търговище (обл. Видин)" appear adjacent in the dropdown — sorted together under "Търговище" — even though they differ in type and oblast.

**Q005 — Sofia raion label format.** For the Sofia raion entry EKATTE 68134-01 ("Средец"), the developer specified "Sofia, р-н Средец" as the settlement_name. By extension, all five Sofia raion entries in dim_settlement (68134-01, 68134-02, 68134-04, 68134-09, 68134-10) follow the pattern "Sofia, р-н {raion_name}" derived from `ek_raion.json`.

### Industry Knowledge

**Finding 1 — Surrogate key stability in SCD Type-1 dimensions (Kimball dimensional modelling).**

In the Kimball data warehouse methodology, surrogate keys are intentionally decoupled from natural keys and must remain stable for the lifetime of a dimension row. Retiring a surrogate key requires updating every fact or bridge table that references it before the delete. In this request, the 30 `dim_store` rows must be remapped before the 15 retired rows are deleted from Supabase. Applicability: directly applicable. Confirms the repair sequence: (1) remap dim_store, (2) upsert dim_settlement + dim_store to Supabase, (3) then delete retired settlement rows via prune_retired_settlements.

**Finding 2 — One-time data repair script vs. full rebuild (data engineering best practice).**

The Kimball Toolkit and Reis & Housley (Fundamentals of Data Engineering) consistently recommend surgical repair scripts over full rebuilds for retrospective dimension corrections when surrogate keys are stable and the defect is confined to a subset of rows. Applicability: the surgical `src/repair_dim_settlement.py` approach is correct. The script now performs a comprehensive one-pass rewrite of the full CSV (adding columns, reformatting all names, removing retired rows) — this remains surgical in the sense that it operates only on the affected dimension and its dependent bridge table.

**Finding 3 — Human-readable administrative type prefixes in geographic data (OGC, NSI conventions).**

OGC standards and OpenStreetMap conventions recommend storing the full administrative type prefix as part of the settlement name when the name alone is ambiguous. The NSI EKATTE registry (`ek_atte.json`) provides the `t_v_m` field that serves exactly this role. Storing type and oblast as separate columns in addition to the qualified display label is consistent with the separation-of-concerns principle used in geographic databases: a display column for human consumption and structured columns for filtering and ordering. Applicability: directly supports the expanded schema design with `settlement_type`, `oblast`, and the three-level sort key in the React app.

**Finding 4 — Client-side sorting of compound labels in React (React Intl / Collator API).**

The MDN documentation and React Intl guidelines recommend using `String.prototype.localeCompare` with a locale tag for Unicode-aware alphabetical sorting of Cyrillic strings. The existing code already uses `localeCompare(b, 'bg')`. Extending the comparator to a three-level (base name → type → oblast) sort maintains Bulgarian-locale awareness at all levels. Applicability: directly applicable to the `getSettlementSortKey()` helper design. The sort key extraction (stripping type prefix and oblast qualifier) is a deterministic string operation that adds negligible overhead at the ~250-row settlement dimension scale.

## Implementation Alternatives

### Alternative 1: Repair Script — Name Label Construction Approach

**Decision context (Task 1):** The `settlement_name` values must be reformatted to "type name (oblast)" for all 246 ek_atte.json-covered rows. Two approaches exist for the base name source.

**Option A — Use ek_atte.json `name` field as base name (authoritative NSI source)**

Construct `new_name = f"{e['t_v_m']} {e['name']} ({e['oblast_name']})"` where `e['name']` is the NSI canonical name from `ek_atte.json`, ignoring the current CSV settlement_name value.

Trade-offs: Prevents double-prefixing for rows like "гр. Попово" that already carry a type prefix; uses authoritative NSI data; normalizes any spelling drift between the CSV and the nomenclature. Codebase impact: minimal — one line in the repair script.

**Option B — Use current CSV `settlement_name` as base name**

Construct `new_name = f"{e['t_v_m']} {row['settlement_name']} ({e['oblast_name']})"` where `row['settlement_name']` is the current value from the CSV.

Trade-offs: Preserves any existing user-facing name customizations; but produces double-prefixed names ("гр. гр. Попово") for the 4 rows already manually corrected. Unacceptable without a prior-correction-strip step.

#### Fork: Name base source for label construction

- Tag: `resolve-autonomously`
- Rationale: Developer's `input.md ## Input` text states "The goal is in dim_settlement the settlement_name to be unique." The ek_atte.json `name` field is the only source that (1) produces the correct unambiguous base name for ALL rows without requiring special-case handling for pre-corrected rows, and (2) is explicitly authoritative per the NSI EKATTE registry which this codebase treats as canonical. Option B has a documented defect (double-prefixing) that makes it unacceptable.

### Alternative 2: Supabase Schema Migration — ADD COLUMN vs. DROP/RECREATE

**Decision context (Task 2):** The existing `dim_settlement` table in Supabase lacks the `settlement_type` and `oblast` columns. They must be added.

**Option A — `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` in `_ENSURE_NULLABLE_DDL`**

Append two `ALTER TABLE IF EXISTS dim_settlement ADD COLUMN IF NOT EXISTS ...` statements to `_ENSURE_NULLABLE_DDL`.

Trade-offs: Idempotent; does not discard existing data; runs on every load_supabase.py invocation as a no-op after the first run. Consistent with the existing pattern for `fact_prices_lookback` column additions in `_ENSURE_NULLABLE_DDL`. Zero downtime. Codebase impact: two lines added to `_ENSURE_NULLABLE_DDL`.

**Option B — Drop and recreate dim_settlement**

Add a DROP + CREATE TABLE sequence in `_MIGRATION_DDL`.

Trade-offs: Guarantees clean schema; but destroys all existing data, requiring a full re-upsert. Violates surrogate key stability. Disruptive to any dependent FK constraints.

#### Fork: Supabase schema migration approach

- Tag: `resolve-autonomously`
- Rationale: `.aib_memory/plan-R-20260518-2256.md ## Constraints` states "Surrogate keys for the 15 canonical settlement rows and all dim_store rows must remain unchanged." A DROP/RECREATE approach violates this constraint. Option A is the only valid choice per the documented constraint. Additionally, the existing `_ENSURE_NULLABLE_DDL` in `src/load_supabase.py` establishes the `ADD COLUMN IF NOT EXISTS` pattern as the workspace-standard migration approach, making Option A consistent with the established convention.

### Alternative 3: React App Settlement Sort Order

**Decision context (Task 3):** Settlement dropdown options must be sorted by base name, then type, then oblast. The base name is not a stored column; it must be derived from the `settlement_name` display label.

**Option A — Extract base name from settlement_name at sort time using settlement_type column**

Add a `getSettlementSortKey(s)` helper that strips the leading `settlement_type + ' '` prefix and trailing ` (обл. ...)` qualifier from `settlement.name`. Handle the "Sofia, р-н {name}" raion pattern as a special case.

Trade-offs: No additional DB column; derives sort key client-side from existing data; handles all patterns with a small deterministic function. Codebase impact: one helper function; sort replacements in 2 functions × 2 paths each = 4 sort call replacements.

**Option B — Add a `sort_name` column to dim_settlement**

Store the bare base name (e.g., "Бяла") as a fourth DB column, returned to the React app for sort comparisons.

Trade-offs: Eliminates client-side string parsing; but requires a fifth DB column; complicates the repair script and DDL migration; adds bandwidth on every dim_settlement fetch; provides marginal benefit at 256-row scale where the string parsing overhead is imperceptible.

#### Fork: Settlement sort key source

- Tag: `resolve-autonomously`
- Rationale: Q004 answer states "The dropdown lists should be ordered by the name, than the type and the area." It does not specify a storage mechanism for the sort key. Option A directly satisfies the requirement with the least additional complexity, using `settlement_type` (already being added) as the type-prefix strip reference. At the 256-row settlement dimension scale, client-side string processing per sort is computationally insignificant. Option B adds unnecessary schema complexity for a one-time sort concern. The developer's answer specifies the behaviour, not the implementation — Option A implements that behaviour correctly.
