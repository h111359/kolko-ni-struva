## Goal

Implement the landing-page flat-table pagination so it no longer calculates or displays total pages, no longer exposes a last-page control, and still lets users go to the first page, previous page, next page, and a specific page number. Keep the change scoped to the current React landing-page flow and its supporting tests and documentation.

## Constraints

- Keep the implementation within the existing React/Vite frontend flow unless a now-unused landing-page count helper can be safely removed without affecting other callers.

- Preserve the row-first loading model in `react-app/src/components/LandingPage.jsx`; the flat-table results must still render without blocking on a total-count query.

- Do not reintroduce any UI element that depends on knowing the last page in advance.

- Keep the user-facing control labels and accessibility labels consistent with the Bulgarian-language UI already used in `react-app/src/components/LandingPage.jsx`.

- Limit documentation updates to files that actually describe the affected pagination behaviour, while always updating `.aib_memory/context.md` to reflect the current product state after implementation.

## Success criteria

- `react-app/src/components/LandingPage.jsx` no longer renders total-page copy such as `от <N>` and no longer renders a last-page button in the flat-table pagination bar.

- `react-app/src/components/LandingPage.jsx` provides a direct specific-page navigation control alongside first, previous, and next navigation.

- The flat-table flow no longer depends on `fetchLandingPageCount` for pagination behaviour, and forward navigation is derived from fetched-row availability instead.

- `react-app/src/components/LandingPage.test.jsx` covers the new pagination behaviour, including the absence of a last-page control and the direct page-jump interaction.

- Running `npm test` in `react-app/` passes after the pagination changes, and any affected product-memory or user-facing documentation is updated.

## Plan

### Task 1: Refactor landing-page pagination state

#### Intent

Remove total-page-dependent pagination state from the flat-table landing-page flow and replace the passive page indicator with direct page-number navigation.

#### Outputs

`react-app/src/components/LandingPage.jsx`

#### Procedure

Update `react-app/src/components/LandingPage.jsx` to remove the `fetchLandingPageCount` import and any `totalCount`, `isCountLoading`, and `totalPages` state or computed values that exist only to calculate total pages for flat-table pagination.

Refactor `react-app/src/components/LandingPage.jsx` so the flat-table pagination bar renders first, previous, specific-page input or submit affordance, and next controls only, with no last-page control and no `Стр. X от Y` text.

Add direct page-jump state handling in `react-app/src/components/LandingPage.jsx` so the user can enter a 1-based page number, convert it to the existing 0-based page index, reject invalid values safely, and trigger `loadRows(buildFilters(), page)` through the existing refresh path.

Keep the forward-navigation rule in `react-app/src/components/LandingPage.jsx` derived from fetched rows and `PAGE_SIZE`, and ensure backward navigation still clamps or disables correctly on the first page.

#### Done criteria

The component no longer references total-page calculation for flat-table pagination, renders no last-page button, and supports first, previous, next, and direct page-number navigation without breaking existing filter and grouping flows.

#### Dependencies

External: none.

#### Risk notes

Direct page-number entry can become inconsistent if the typed value is not synchronized with `currentPage`, so the control state must update whenever a new page loads.
