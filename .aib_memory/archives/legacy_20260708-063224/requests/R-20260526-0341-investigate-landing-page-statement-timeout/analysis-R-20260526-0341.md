## Research Results

### Workspace pattern-scan findings

- `react-app/src/components/LandingPage.jsx` already decouples `fetchLandingPageCount()` from the immediate row render path, so the remaining failure is now attributable to `fetchLandingPageRows()` and its backing `get_landing_page_rows` RPC rather than to total-count retrieval.

- `src/load_supabase.py` implements `get_landing_page_rows` as a dynamic SQL query that joins `fact_prices_lookback` to `dim_product`, `dim_category`, `dim_store`, `dim_company`, `dim_settlement`, and `dim_file`, computes effective price inline, applies optional substring filtering on `dp.product_name`, orders by `dp.product_name` and `dstore.store_name`, and paginates with `OFFSET ... LIMIT ...`. This keeps the query functionally complete but makes the current page fetch vulnerable to expensive joins, sorts, and skipped-row computation on a large dataset.

- The current index set in `src/load_supabase.py` primarily covers fact-table foreign keys and date/category combinations. There is no dedicated support for the substring product-name filter, no ordering-aware landing-page row index, and no explicit page-key-first query structure.

- `react-app/src/components/LandingPage.test.jsx` and `react-app/src/lib/dataService.test.js` define the current contract as a rows-only RPC for `fetchLandingPageRows()`, asynchronous count loading, and standard offset pagination. They do not yet encode a thinner two-step row fetch strategy, stable tie-break ordering, or stale-count protection.

### Industry knowledge

- PostgreSQL documents that rows skipped by `OFFSET` still have to be computed inside the server. Applicability: the current landing-page flat-row query will get progressively more expensive for deeper pages unless the query is restructured to avoid broad skipped-row work.

- PostgreSQL documents that `pg_trgm` GiST and GIN indexes support fast `LIKE` and `ILIKE` searches, including non-left-anchored patterns such as `%term%`. Applicability: preserving current substring product search semantics likely requires trigram support or an equivalent alternative if product-name filtering remains part of the landing-page path.

- Supabase guidance on Postgres indexing emphasizes targeted indexes, ordering-aware indexes, and advisor-driven query tuning for large datasets. Applicability: the chosen remediation path should either align the row query with existing indexes or add only the focused indexes needed for the preserved UX contract.

### AI Agent critique

- `.aib_memory/context.md` is stale relative to the current code because it still states that `LandingPage.jsx` calls `fetchLandingPageRows` and `fetchLandingPageCount` in parallel via `Promise.all`, while the current component loads rows first and count asynchronously afterward.

- `react-app/src/components/LandingPage.jsx` still contains a stale header comment describing the old parallel-fetch behavior.

- `react-app/src/lib/dataService.js` still contains stale documentation for `fetchLandingPageCount()` that says it is called in parallel with `fetchLandingPageRows()`.

- `react-app/src/components/LandingPage.jsx` issues asynchronous count requests without request-version guarding, so an older count response can still overwrite a newer filter state.

- `src/load_supabase.py` orders landing-page rows by `product_name` and `store_name` only, which is not a fully unique order and can produce unstable pagination boundaries when duplicate values exist.

- `src/load_supabase.py` keeps the flat-row RPC fully dimension-joined even when only a subset of those joins are needed to identify the current page rows.

- `react-app/src/components/LandingPage.test.jsx` does not cover race conditions introduced by deferred count loading, which leaves a regression gap for the current UI state management.
