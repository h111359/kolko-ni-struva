## Research Results

### Workspace Pattern Scan

- The React landing page mounts and immediately calls `triggerRefresh()` from `react-app/src/components/LandingPage.jsx`, which in turn runs `loadOptions()` and `loadRows()` concurrently for the default first-page state. The default state includes the most recent date and no narrowing filters, which means the first `fetchLandingPageRows()` call exercises the widest anonymous query path.

- `react-app/src/lib/dataService.js` sends `fetchLandingPageRows()` to the `get_landing_page_rows` RPC with `p_date_key`, `p_offset`, and `p_limit`, and no longer depends on `fetchLandingPageCount()` for paginator rendering. The browser failure reported by the developer is therefore tied to the rows RPC itself, not to the previously separated count RPC.

- `src/load_supabase.py` defines `get_landing_page_rows()` as dynamic SQL that still builds the visible page by joining `fact_prices_lookback` to `dim_product` and `dim_store`, applying optional filters, and then ordering by `dp.product_name`, `dstore.store_name`, `f.product_key`, `f.store_key`, and `f.file_key` before `OFFSET` and `LIMIT` are applied. On the default first-page load this means the database must produce a globally sorted joined result for the selected date slice before it can return the first 100 rows.

- The existing index set in `src/load_supabase.py` is strong for date, store, category, and product-name substring filtering: there are B-tree indexes on `fact_prices_lookback(date_key)`, `fact_prices_lookback(date_key, store_key)`, `fact_prices_lookback(date_key, category_key)`, and `fact_prices_lookback(date_key, store_key, category_key)`, plus `pg_trgm` and sort-support indexes on `dim_product.product_name` and `dim_store.store_name`. Those help filtered paths, but they do not create an obvious index-friendly access path for the unfiltered default ordering across joined dimension text columns.

- The current row RPC already reduced one earlier cost by removing the `COUNT(*) OVER()` window count from the row-returning query. The remaining timeout therefore indicates that the default first-page query is still too expensive even without the count work, which points to the joined global sort and wide pre-limit row shaping as the dominant remaining cost.

- `react-app/src/components/LandingPage.test.jsx`, `react-app/src/lib/dataService.test.js`, and `tests/test_load_supabase.py` all use mocks rather than a live Supabase database. The local frontend test suite passes and the frontend production build succeeds, which demonstrates compile health and mocked contract stability, but it does not prove that the anonymous live SQL plan now fits within Supabase timeout limits.

- The product documentation and context indicate that the frontend runs directly against Supabase with an anon or publishable key. That makes the live timeout budget part of the production contract, not a local-only development nuisance.

### Industry Knowledge

1. Supabase documents that the default `anon` role statement timeout is 3 seconds, and that browser-originated client-library requests use that role unless the project is reconfigured. Applicability: the landing-page first-page load must be designed to complete under the anonymous runtime budget instead of depending on a long-running query budget that is unavailable to the current client-only architecture.

2. PostgreSQL documents that B-tree indexes are effective for equality, range predicates, and sorted retrieval when the query ordering matches indexed columns. Applicability: the current default row query orders on joined dimension text columns rather than on a fact-local indexed key path, so the planner has weaker opportunities to satisfy the first-page request through an index-driven ordered scan.

3. PostgreSQL documents that `pg_trgm` indexes accelerate `LIKE` and `ILIKE` searches, including non-left-anchored substring matching. Applicability: the existing trigram index on `dim_product.product_name` is appropriate for the optional product-name filter, but it does not help the default failing case because the first-page load usually runs with `p_product_name = NULL` and therefore never activates that index path.

### Decision Points

#### Decision Point: Verification evidence required before closure

- Applies to: the final acceptance step for the bug fix, where the agent must decide what level of proof is required before claiming the issue is resolved.

- Why alternatives exist: the repository offers fast local checks such as mocked unit tests and a production build, but the reported failure happens in the live anonymous Supabase path and earlier attempts were declared complete without fixing the real site behavior.

- Alternative A: Mocked automated checks only.
  - Description: rely on Vitest unit tests and static build success as the closure signal.
  - Trade-offs: fast and repeatable, but incapable of proving the live Supabase query no longer times out.
  - Expected codebase impact: no additional runtime validation workflow beyond existing test commands.

- Alternative B: Build plus live anonymous-path verification.
  - Description: require the fix to be validated in the real frontend-to-Supabase execution path after provisioning any SQL changes.
  - Trade-offs: slower and environment-dependent, but directly tests the failing production behavior.
  - Expected codebase impact: may require stronger validation steps and documentation of the live verification workflow.

- Tag: resolved-by-user

- Chosen alternative: Alternative B — Build plus live anonymous-path verification.

- Rationale/Resolution: `.aib_memory/input.md` explicitly says, `Test rigorously as it is alredy more than 5 attempts to find and fix the issue, you are telling me that everything is fixed and the site is still not working. It is critical to be sure that the site is working and no failures happen`. That instruction unambiguously rules out mock-only proof.

#### Decision Point: Primary mitigation strategy for the first-page timeout

- Applies to: the backend query path used by `fetchLandingPageRows()` during the landing page's default first-page load.

- Why alternatives exist: the timeout could be approached by relaxing database timeout settings, by preserving the current SQL and only improving test coverage, or by reshaping the row RPC so the default first-page query does less work before pagination is applied.

- Alternative A: Raise timeout budgets for the anonymous query path.
  - Description: change role-level or function-level timeout settings so the current row query is allowed to run longer.
  - Trade-offs: potentially quick to apply, but it does not remove the underlying expensive execution path and weakens reliability under shared compute pressure.
  - Expected codebase impact: SQL function DDL or database role configuration changes without materially simplifying the row query.

- Alternative B: Keep the current SQL shape and add stronger tests only.
  - Description: leave `get_landing_page_rows()` structurally unchanged and focus on better validation.
  - Trade-offs: improves confidence reporting, but cannot eliminate the live timeout because the failing query plan is untouched.
  - Expected codebase impact: test-only changes while the live first-page failure remains likely.

- Alternative C: Refactor `get_landing_page_rows()` so the visible page is selected through a thinner, more index-friendly slice before expensive joins and wide row shaping.
  - Description: restructure the RPC to identify the requested page keys first, then join and enrich only those visible rows while preserving deterministic ordering.
  - Trade-offs: more implementation work, but it addresses the remaining expensive work in the real failing query path rather than hiding it behind a larger timeout.
  - Expected codebase impact: updates to `src/load_supabase.py`, potential contract-alignment work in `react-app/src/lib/dataService.js`, and stronger SQL-shape tests.

- Tag: resolve-autonomously

- Chosen alternative: Alternative C — Refactor `get_landing_page_rows()` so the visible page is selected through a thinner, more index-friendly slice before expensive joins and wide row shaping.

- Rationale/Resolution: `.aib_memory/input.md` requires the agent to `Find why. Fix the issue.` and to be `sure that the site is working and no failures happen`. In the current workspace evidence, Alternatives A and B do not satisfy that requirement because they either preserve the expensive query plan or avoid changing it entirely. The request therefore resolves toward the only alternative that directly addresses the observed remaining cause in the live failing path.
