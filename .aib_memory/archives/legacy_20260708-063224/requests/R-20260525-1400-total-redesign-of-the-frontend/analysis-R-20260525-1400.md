# Analysis: R-20260525-1400

## Overview

**Request ID:** R-20260525-1400

**Request title:** Total redesign of the frontend

### Background

The current React Analytics App exposes price data across five distinct navigation pages — Начало (Home), Цени по категория (Report 1), Продукти (Report 2), Сравнение по места (Report 3), and Файлове (Files) — each requiring the user to navigate to the appropriate page and configure its specific filters independently. The Files page imposes an additional two-step interaction: first select a source CSV file from a summary list, then drill down into the detail table for that file. The developer requests a total redesign that removes all four non-Files pages, consolidates all interaction onto a single unified landing page derived from the existing Files detail table, and adds comprehensive cross-dimensional filters, a product name text search, a price range numeric filter, and a two-level grouping capability with aggregated metrics. The redesign also renames price columns to clarify the distinction between the regular retail price and the effective promotional price.

### Scope

## Input Interpretation

The developer requests a complete redesign of the React Analytics App front end. The current application presents five navigation pages (Начало, Цени по категория, Продукти, Сравнение по места, Файлове) with the Files page acting as a two-step drill-down: first a list of source CSV files, then a paginated detail table of individual `fact_prices_lookback` rows for the selected file. This structure is to be replaced by a single unified landing page.

The new landing page shall:

1. Display `fact_prices_lookback` rows for the most recently loaded date immediately on page load, paginated, without requiring the user to first select a source file. Data loading uses a generic Supabase PostgreSQL RPC (`get_landing_page_rows`) that handles all filtering, enrichment, and pagination server-side (Q001: Option B resolved by user).

2. Show the introductory text from the current "Начало" page (as defined in HomePage.jsx) above the filters section of the landing page.

3. Present the detail table with the following columns in ungrouped (flat detail) view:
   - Файл (file name from `dim_file`, resolved server-side in the RPC)
   - Продукт
   - Категория
   - Населено място
   - Магазин
   - Верига
   - Цена без промоция (= `retail_price`)
   - Цена (= `min(retail_price, promo_price)` when `promo_price > 0`, otherwise `retail_price`; equivalent to the existing `calculatePrice()` function in `dataService.js`)

   The "Промо цена" (raw `promo_price`) column is not shown in ungrouped view but appears as an aggregation metric target in grouped view.

   Lookback columns (retail_price_day1, promo_price_day1, retail_price_day2, promo_price_day2) are removed from the display entirely.

4. Provide five dropdown filters — Населено място, Категория, Верига, Магазин, and Дата — with full multi-dimensional cross-filtering: each dropdown is restricted to only options that have data for the current combination of the other four active filters. This mirrors and extends the bidirectional pattern already present in the now-deleted Report2.jsx. The Дата filter pre-selects the most recently available `date_key`. (Q002: Option C resolved by user.)

5. Provide a product name text filter: a case-insensitive substring match against `product_name` from `dim_product`, handled server-side within the main RPC.

   - For "Цена" (effective price = `calculatePrice(row)`): average, minimum, maximum across all matching rows in the group
   - For "Промо цена" (raw `promo_price`, counting only rows where `promo_price > 0`): average, minimum, maximum

9. The App header navigation bar (five nav buttons) and the global date selector are removed. The header retains only the app title and subtitle.

10. The following components and their test files are deleted:

## Research Results

### Workspace Pattern-Scan Findings

- **fetchAllFileRows filters by file_key — not by date_key**: The function `fetchAllFileRows` in `dataService.js` (line 1299) fetches `fact_prices_lookback` rows filtered by `file_key`. The new RPC `get_landing_page_rows` filters by `date_key` (and optionally other dimension keys) and returns enriched rows directly from PostgreSQL, eliminating the need for a `fetchAllFileRows`-style client-loop.

- **file_key / file_name enrichment in RPC**: The new `get_landing_page_rows` RPC must JOIN `dim_file` and return `file_name` as a labelled column so the "Файл" column in the detail table is populated without a client-side lookup. The `dimensions.files` cache remains available as a fallback.

- **calculatePrice() alignment**: The new "Цена" column definition matches the existing `calculatePrice()` function in `dataService.js`. In the RPC this is expressed as `LEAST(f.retail_price, NULLIF(f.promo_price, 0))` with a `COALESCE` fallback to `f.retail_price`. The client-side `calculatePrice()` is retained for tests and for the grouping aggregation boundary checks.

- **Cross-filter RPC pattern exists in codebase**: Report2.jsx uses `fetchCategoriesForSettlement`, `fetchSettlementsForCategory`, and `fetchSettlementsForDate` calling the RPCs `get_categories_for_settlement`, `get_settlements_for_category`, and `get_settlements_for_date`. The same pattern is extended to cover all five dimensions with five new `get_lp_options_*` RPCs (one per dimension). Each RPC executes a `SELECT DISTINCT <key>, <label> FROM fact_prices_lookback JOIN dim_* WHERE <other_filter_conditions>` query.

- **Global date selector becomes orphaned**: The `<select id="date-selector">` in App.jsx passes `selectedDate` to child page components. After the redesign, the App header date selector is removed entirely. `LandingPage.jsx` manages its own `selectedDate` state via the "Дата" filter dropdown.

- **Dimensions cache adequacy**: `fetchDimensions()` loads all six dimension tables into `_dims`. This cache is still needed for populating initial dropdown option lists before the first cross-filter RPC call, and for the `dates` array used to pre-select the most recent date.

### Industry Knowledge

1. **Server-side pagination for large datasets (Tanstack Table v8 / AG Grid documentation)**: For `fact_prices_lookback`, a single date contains approximately 1.3 million rows. The chosen Option B (generic RPC) delivers server-side pagination naturally: the RPC accepts `p_offset` and `p_limit` parameters and returns a `rows` array plus a `total_count`, keeping memory consumption constant. **Applicability: HIGH** — confirmed by developer choice.

2. **Full multi-dimensional cross-filtering (Nielsen Norman Group / Supabase PostgREST patterns)**: Option C cross-filtering requires five additional RPCs, each executing a DISTINCT JOIN on `fact_prices_lookback` scoped to the current filter combination. Supabase free-tier query timeouts (~3s) are a risk for unindexed cross-filter queries on a 1.3M-row table. Existing B-tree indexes must be verified before implementation. **Applicability: HIGH** — confirmed by developer choice; index verification is a pre-implementation blocker.

3. **Supabase RPC for GROUP BY aggregations**: Supabase PostgREST does not expose GROUP BY in the client query builder. Grouped aggregations (avg, min, max) must be implemented as PostgreSQL functions. The new `get_landing_page_grouped` RPC accepts grouping dimension names as `TEXT` parameters and uses dynamic SQL (`EXECUTE format(...)`) to construct the GROUP BY clause. **Applicability: HIGH** — required by chosen approach.

4. **Decimal input validation in Bulgarian locale (MDN Web Docs / WHATWG HTML Living Standard)**: `<input type="number">` does not support `,` as the decimal separator in Chrome/Firefox. `<input type="text">` with a validation regex `/^\d+([.,]\d{1,2})?$/` is the correct approach for Bulgarian locale. Invalid input triggers an inline error; the filter is not applied until a valid value is entered. **Applicability: MEDIUM** — confirmed in scope.

5. **Dynamic SQL security in PostgreSQL (PostgreSQL 15 documentation)**: The `get_landing_page_grouped` RPC uses `EXECUTE format()` with user-supplied grouping dimension names. Names must be validated against a whitelist (`ARRAY['settlement_name','category_name','company_name','store_name','date_key']`) before being interpolated with `%I` (identifier quoting) to prevent SQL injection. **Applicability: HIGH** — security-critical requirement for the grouping RPC.

## Decision Register

### Decision Points

#### Decision Point: Data Loading Strategy for Date-Scoped Flat Detail View

**Tag:** `resolved-by-user`

**Chosen alternative:** Option B — Single generic Supabase PostgreSQL RPC (`get_landing_page_rows`) accepting all filter and pagination parameters; all filtering, enrichment, and pagination handled server-side in SQL.

**Resolution outcome:**

- A new PostgreSQL function `get_landing_page_rows(p_date_key INT, p_settlement_key INT, p_category_key INT, p_company_key INT, p_store_key INT, p_product_name TEXT, p_price_min NUMERIC, p_price_max NUMERIC, p_offset INT, p_limit INT)` is added to `src/load_supabase.py` and provisioned via `python src/load_supabase.py`

- The function returns a JSON array of enriched rows (dimension labels resolved in SQL via JOINs) plus a `total_count` integer for pagination

- A companion function `get_landing_page_rows_count(...)` with the same filter parameters returns only the count, used for pagination indicator display

- `dataService.js` exposes `fetchLandingPageRows(filters, page, pageSize)` and `fetchLandingPageCount(filters)` wrappers

- `LandingPage.jsx` triggers a new RPC call on every filter change, page change, or date change; client-side dimension cache is used only for populating dropdown option lists

**Rationale:** Developer selected Option B. The RPC approach cleanly resolves the product-name substring filter challenge (PostgREST `.ilike()` cannot join across tables without an RPC), delivers row enrichment in SQL rather than client-side loops, and is consistent with the workspace's established "Report-query pushdown" principle (context.md).

#### Decision Point: Cross-Filtering Between Dropdown Filters

**Tag:** `resolved-by-user`

**Chosen alternative:** Option C — Full multi-dimensional cross-filtering across all five dropdowns; each dropdown shows only options that have data for the current combination of the other four active filters.

**Resolution outcome:**

- Five new PostgreSQL functions added to `src/load_supabase.py`:
  - `get_lp_options_settlement(p_date_key, p_category_key, p_company_key, p_store_key)` → available settlement options
  - `get_lp_options_category(p_date_key, p_settlement_key, p_company_key, p_store_key)` → available category options
  - `get_lp_options_company(p_date_key, p_settlement_key, p_category_key, p_store_key)` → available company options
  - `get_lp_options_store(p_date_key, p_settlement_key, p_category_key, p_company_key)` → available store options
  - `get_lp_options_date(p_settlement_key, p_category_key, p_company_key, p_store_key)` → available date options

- Each function executes `SELECT DISTINCT <key>, <label> FROM fact_prices_lookback JOIN dim_* WHERE <other_filter_conditions>`

- `dataService.js` exposes `fetchLandingPageOptions(dimension, currentFilters)` that dispatches to the appropriate RPC

- `LandingPage.jsx` re-fetches all five option lists on every filter change; if a currently selected value is no longer in the updated option list, that selection is cleared

- Index coverage on `fact_prices_lookback` for `date_key`, `settlement_key`, `category_key`, `store_key` must be verified before implementation

**Rationale:** Developer selected Option C. Maximizes usability — no "empty results" from invalid filter combinations, and the Магазин dropdown is always constrained to a practical count. Extends the existing bidirectional cross-filter pattern from Report2 to all five dimensions.
