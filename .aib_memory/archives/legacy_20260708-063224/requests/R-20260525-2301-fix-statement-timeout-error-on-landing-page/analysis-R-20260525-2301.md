# Analysis

## 1. Overview
- Request ID: R-20260525-2301
- Request title: Fix statement timeout error on landing page

### Background
The landing page triggers a statement timeout when calling `fetchLandingPageCount`. A previous attempt extracted the `COUNT(*) OVER()` out of the data row query into a separate RPC `get_landing_page_count`, but the dedicated count query itself is still timing out.

### Scope
- Investigate and fix the performance issue in `get_landing_page_count` in `src/load_supabase.py` and `react-app/src/lib/dataService.js`.
- Provide a robust way to avoid the timeout, either by removing the timeout-prone `COUNT(*)` execution over massive rows, or by optimizing the SQL query.
- Ensure the frontend pagination and UI do not crash due to timeout.

### Out of scope
- Removing pagination from the data tables entirely.
- Restructuring the dimension tables or fact tables.

## 3. Input Interpretation
The developer reports that `fetchLandingPageCount` triggers a "canceling statement due to statement timeout" error when the Landing Page loads. Since no data is displayed as a result, the application UI is functionally broken. The timeout is likely caused by the query definition of `get_landing_page_count` performing expensive operations like joining multiple dimensions mapping to the whole fact table with many `OR IS NULL` filters, even when filters are minimal. 
The developer explicitly selected to resolve this by extracting exact counts with dynamic SQL logic in the RPC.

## 4. Research Results
- The RPC `get_landing_page_count` in `src/load_supabase.py` explicitly does a `SELECT COUNT(*)` on `fact_prices_lookback f` while always doing `INNER JOIN dim_product dp` and `INNER JOIN dim_store dstore`.
- If only `p_date_key` is provided (as is on Landing Page load), Postgres still evaluates the join logic across a massive number of rows, and checks all the `OR NULL` conditions.
- Additionally, `p_price_min` and `p_price_max` logic uses complex `COALESCE` and `LEAST` functions, which evaluated over million of rows is slow.
- The `fact_prices_lookback` relation has indexes, but the unconditional joins combined with `OR IS NULL` conditional filters prevent the query planner from using simple index-only scans for counts (the "catch-all" query pattern).
- Since this is an SQL RPC, dynamic SQL (`EXECUTE`) can be used to build the query dynamically and only include joins and where clauses when filters are actively provided.

## 5. Decision Register

### Decision Points

#### Decision Point: How to resolve the count statement timeout
The statement timeout is caused by querying `get_landing_page_count` across large swaths of rows with complex conditionals.

- Extract exact counts with dynamic SQL logic.
  - Trade-offs: Bypasses the inefficient Postgres execution plan for static queries with many `OR IS NULL` conditionals. More verbose PL/pgSQL.
  - Expected codebase impact: Modifies `src/load_supabase.py` implementation of the RPC `get_landing_page_count`.

- **Tag**: `resolved-by-user`
- **Resolution**: The user chose to extract exact counts using dynamic SQL logic in the RPC.
