## Overview

- Request ID: R-20260612-0710

- Request title: Reduce WAL growth during Supabase load

### Background

Running `src/load_supabase.py` on the current dataset (approximately 1.37 million rows in `fact_prices_lookback`) generates enough WAL to push the managed Supabase free-tier PostgreSQL instance into read-only mode. The developer requested changes to the Supabase sync path that reduce WAL pressure while preserving the same analytical result visible to frontend consumers. Three Q-blocks were issued in the prior analysis run to resolve the key architectural choices. The developer has now answered all three, resolving the fact-table load strategy, the projection-refresh strategy, and the provisioning-cadence model.

### Scope

- `src/load_supabase.py`: replace `insert_lookback()` TRUNCATE + batched INSERT with TRUNCATE + COPY FROM STDIN; replace the `_CREATE_DDL` projection table definition with a VIEW definition; add the projection table drop to `_MIGRATION_DDL`; remove `refresh_landing_page_projection()` from the sync flow and from the module; remove projection-specific indexes from `_CREATE_INDEXES`; remove dead module-level constants.
- `tests/test_load_supabase.py`: update tests that assert physical projection table behavior; remove two pre-existing failing test assertions; add tests for COPY-based load behavior.
- `README.md`: update sync documentation to describe the COPY-based fact load and the view-based projection.

### Out of scope

Changes to the React application, `src/extract.py`, `src/transform.py`, and `menu.py` are excluded. Optimizing `upsert_dim()` with a row-difference guard is a potential future improvement but is not part of this request. Supabase platform-tier changes and unmanaged database administration are excluded.

1. The fact-table load strategy for `fact_prices_lookback` is COPY-based full replacement: replace the `execute_batch_rows()` INSERT loop in `insert_lookback()` with a `cur.copy_expert()` call that streams the existing CSV directly through `COPY … FROM STDIN WITH (FORMAT CSV, HEADER, NULL '')`, preserving the empty-string-to-NULL coercion behavior of the prior `_coerce()` path without application-level transformation.

3. `src/load_supabase.py` must remain a single self-provisioning sync command: `create_tables()` stays part of every run; no split provisioning/data-sync modes are introduced.

## Research Results

### Workspace pattern-scan findings

- `insert_lookback()` in `src/load_supabase.py` currently uses `TRUNCATE` then `execute_batch_rows()` with batched `INSERT` statements. The column list for the COPY target is already enumerated in the function as `columns`. The COPY variant can reuse the same ordered column list and rely on `COPY … WITH (FORMAT CSV, HEADER, NULL '')` to replicate the empty-string-to-NULL coercion performed by `_coerce()`, eliminating the per-row Python transformation pass.

- `_REFRESH_LANDING_PAGE_PROJECTION_SQL` and `_REFRESH_LANDING_PAGE_PROJECTION_BATCH_SQL` define the full denormalized SELECT that currently populates the physical projection table. This SELECT body (`FROM fact_prices_lookback f JOIN dim_product … JOIN dim_category … JOIN dim_store … JOIN dim_company … JOIN dim_settlement … LEFT JOIN dim_file …`) is the exact body needed for `CREATE OR REPLACE VIEW landing_page_row_projection AS SELECT …`.

- `_CREATE_DDL` defines `landing_page_row_projection` as a `CREATE TABLE IF NOT EXISTS` block at the end of the DDL string. That block must be replaced with a `CREATE OR REPLACE VIEW` statement.

- `_MIGRATION_DDL` already drops legacy tables before `_CREATE_DDL` runs. Adding `DROP TABLE IF EXISTS landing_page_row_projection CASCADE;` to this block ensures that an existing physical projection table is dropped before the view creation attempt, allowing `CREATE OR REPLACE VIEW` to succeed on existing Supabase databases. `CASCADE` is required to also drop the physical table's dependent indexes.

- `_CREATE_INDEXES` defines three indexes that target the physical projection table: `idx_lp_row_projection_page`, `idx_lp_row_projection_filter_keys`, and `idx_lp_row_projection_product_name_trgm`. PostgreSQL does not allow indexes on regular views. These three index definitions must be removed.

- `refresh_landing_page_projection()` and its call in `main()` step 7 become dead code once the physical table is replaced with a view. Both must be removed.

- The `GRANT SELECT ON {LANDING_PAGE_ROW_PROJECTION} TO anon;` statement in `_CREATE_RPC_FUNCTIONS` is valid for views and does not need to change.

- `get_landing_page_grouped()` in `_CREATE_RPC_FUNCTIONS` reads directly from `fact_prices_lookback` with explicit JOINs, not from `landing_page_row_projection`. It is unaffected by this change.

- `get_landing_page_rows()` and all five `get_lp_options_*` RPCs reference `{LANDING_PAGE_ROW_PROJECTION}` by name. After the view is created, those RPCs resolve the same name and produce equivalent result sets at query-time join cost instead of projection-table read cost.

- `tests/test_load_supabase.py` contains the following test cases that require changes:
  - `test_create_ddl_contains_landing_page_projection` — asserts `CREATE TABLE IF NOT EXISTS {LANDING_PAGE_ROW_PROJECTION}` in `_CREATE_DDL`; must be updated to assert VIEW definition.
  - `test_index_ddl_contains_landing_page_projection_indexes` — asserts the three projection indexes; must be updated to assert their absence.
  - `TestLandingPageProjection.test_refresh_landing_page_projection_executes_refresh_and_commits` — validates the TRUNCATE+INSERT behavior of the removed function; must be removed or replaced.
  - `test_index_ddl_contains_lookback_composite_date_store_index` — asserts `idx_fact_prices_lookback_date_store` which does not exist in `_CREATE_INDEXES`. Pre-existing failure; must be removed.
  - `test_index_ddl_contains_report_slice_index` — asserts `idx_fpl_date_store_category` which does not exist in `_CREATE_INDEXES`. Pre-existing failure; must be removed.

- `test_executes_five_ddl_statements` asserts exactly 5 DDL execute calls in `create_tables()`. The count remains 5 (_MIGRATION_DDL, _CREATE_DDL, _ENSURE_NULLABLE_DDL, _CREATE_RPC_FUNCTIONS, _CREATE_INDEXES) and does not change with this request.

### Industry knowledge

- **PostgreSQL COPY command performance**: PostgreSQL documentation states that `COPY` bypasses most per-row overhead of individual `INSERT` statements because it processes input in bulk without re-evaluating constraint triggers per row. For large tables, `COPY` regularly achieves materially lower WAL volume than batched `INSERT` because the server writes pages rather than individual tuple entries in the WAL stream (PostgreSQL 15 docs §14.4 "Populating a Database"). Applicability: directly applicable. `psycopg2` exposes `cursor.copy_expert()` for `COPY … FROM STDIN` and the current codebase already reads the CSV file from disk, making the switch low-risk and non-invasive to the surrounding transaction structure.

- **PostgreSQL regular views vs. physical tables for WAL**: A PostgreSQL regular view is a stored query with no materialized storage. It generates zero WAL because it stores no data. Reads expand the view definition at query time and rely on indexes on the underlying tables. The existing `fact_prices_lookback` indexes (`idx_fpl_category_key`, `idx_fpl_settlement_key`, `idx_fpl_product_key`, `idx_fpl_file_key`, `idx_fact_prices_lookback_date_key`) and `idx_dim_product_name_trgm` on `dim_product` remain in place and will be available to query plans that access the view. Applicability: high; trading write-time cost (zero WAL) for read-time cost (join execution each query) is the correct tradeoff for a rarely-written, frequently-read analytics projection on a WAL-constrained instance.

- **Supabase WAL budget on free tier**: Supabase free tier operates with a WAL-archiving configuration that, when write volume exceeds budgeted thresholds in a single session, results in the instance temporarily entering `transaction_read_only = on` (Supabase documentation, "Database — Overview"). Supabase recommends COPY for large data operations over direct connections (Supabase docs, "Connecting to Postgres directly"). Applicability: directly relevant; the described failure mode matches the current symptom, and the documented mitigation aligns with both chosen approaches.

- **psycopg2 COPY integration**: `psycopg2` provides `cursor.copy_expert(sql, file)` which maps directly to the PostgreSQL `COPY … FROM STDIN WITH (…)` server command. Combined with the `NULL ''` COPY option, empty CSV cells are interpreted as SQL NULL without any application-level transformation, replacing the `_coerce()` function's behavior at the server side. `cur.rowcount` after `copy_expert` returns the number of rows copied on psycopg2 ≥ 2.4; on older versions it may return -1. Applicability: high; the existing `csv_path` argument in `insert_lookback()` can be opened directly and passed to `copy_expert`, requiring no intermediate buffer.

- **View security in Supabase / PostgREST**: `GRANT SELECT ON view TO anon` is valid for PostgreSQL views and is sufficient for PostgREST to expose the view as a queryable endpoint. The security-definer behavior of the `get_landing_page_rows` and `get_lp_options_*` RPCs means the view is accessed through the function's security context, so no additional RLS policy changes are required. Applicability: medium; no security model changes are needed beyond ensuring the existing GRANT statement is retained.
