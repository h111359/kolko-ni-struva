## Executive Summary
- Request ID: `R-20260509-2012`.

- Request title: Add SQL query log page for current session.

- High-level purpose: provide an in-app debugging page that shows the database-query activity generated by the current React app session so incorrect visualized data can be investigated from concrete request evidence.

- The controlling code path is the React frontend, especially `react-app/src/App.jsx`, `react-app/src/lib/dataService.js`, and `react-app/src/lib/supabase.js`, because the app is client-only and queries Supabase directly through the browser.

- Local research shows the app currently has no dedicated observability surface: startup loads dimensions through `fetchDimensions()`, and report pages trigger additional shared data-service queries and RPC calls.

- The key architectural constraint is that the browser-side Supabase client expresses table reads and RPC calls through PostgREST-style requests; exact backend SQL text is not naturally available in the existing frontend code path.

- This creates one significant decision for implementation: whether the new page should show practical client-visible request intent or true database-side SQL captured through database observability tooling.

- This analysis run populated or updated these `request.md` sections: `## Assumptions`, `## Plan`, `## Documentation`, `## Questions & Decisions`, `## Code and Asset Scan for Impacted Components`, and `## Internal Review of Request and Product Docs`.

## Domain Knowledge Essentials
- Visualization debugging: the business need behind this request is not generic observability but diagnosing why a rendered chart or table in the public app looks wrong.

- Session query log: a temporary record of query activity generated while the current browser app session is running. It is relevant because the user wants to inspect what happened during “this session,” not build a full audit warehouse.

- Supabase query: in this product, a browser-triggered request against a table, view, or RPC function exposed by Supabase/PostgREST. This is the app’s practical unit of data access.

- SQL statement: a backend database statement such as `SELECT ...` or `CALL ...`. It matters here because the request uses the phrase “SQL queries,” which may or may not align with what the browser can observe directly.

- Impacted roles:
  - End user or analyst debugging a suspicious visualization in the React app.
  - Developer maintaining the React data layer and report components.
  - Operator maintaining Supabase configuration if database-side auditing is chosen.

- Business processes touched:
  - App startup dimension loading.
  - Report 1 settlement and category reads.
  - Report 2 products and cross-filter RPC reads.
  - Report 3 category-driven reads.
  - User-led investigation of incorrect visual results.

- Acceptance impact from a product perspective:
  - The new page must make the data-fetch path more transparent.
  - The output must be understandable enough to help explain a suspicious visualization.
  - The page must not create false confidence by claiming to show exact SQL if it only shows client-visible request metadata.

## Technical Knowledge & Terms
- React SPA: a single-page application built with React and Vite. In this workspace, the React app runs entirely in the browser and has no custom backend.

- Supabase JS client: the `@supabase/supabase-js` library used by the app to query database tables and RPC functions from the browser.

- PostgREST: the REST layer behind Supabase table and RPC access. It exposes database resources as HTTP requests rather than requiring handwritten SQL in frontend code.

- RPC: a remote procedure call against a PostgreSQL function exposed through the `/rpc` path. In this app, examples include `get_available_dates`, `get_settlements_for_date`, `get_categories_for_settlement`, and `get_settlements_for_category`.

- Session store: an in-memory application state holder that survives page navigation within the app but not necessarily a full reload. This is the most natural fit for “current session” in the existing frontend architecture.

- Non-functional considerations:
  - Reliability: logging must not change existing report query semantics.
  - Performance: instrumentation should add only small overhead because some report queries already paginate large result sets.
  - Security: the page must not leak credentials or unnecessarily preserve sensitive parameters.
  - Operations: if true SQL capture is required, the solution may extend beyond the frontend and involve Supabase-side observability.

## Research Results
- Pattern scan against current workspace implementation:
  - The React app already centralizes data access better than many component-heavy SPAs by routing most reads through `react-app/src/lib/dataService.js`. This is favorable for adding a session query log without widespread component rewrites.
  - Startup behavior is split between `App.jsx` and the shared data layer, so a query log that starts only after entering a report page would miss one of the most important debugging phases.
  - Existing test structure already mocks the Supabase client and data-service functions. That makes automated validation of a logging layer practical without live network calls.

- Pattern scan against request scope:
  - The user’s goal is investigative, not merely decorative. The page therefore needs operator-useful fields such as source function or page, query target, parameters, timing, and success state.
  - The word “SQL” is the main ambiguity. In this architecture, the frontend issues Supabase query-builder and RPC calls, which represent query intent but not always the exact backend SQL statement.
  - Because the app is public and client-only, any design that depends on privileged database logs would materially widen scope and operational dependency.

- Explicit risks:
  - A frontend-only implementation could disappoint the user if it is labeled as exact SQL while actually showing request metadata.
  - A database-side implementation could exceed the scope of “add additional page” by requiring Supabase extensions, log access, or role-level configuration.
  - Incomplete instrumentation would produce misleading logs and undermine debugging trust.

## External Benchmarking
- Benchmark reference: PostgREST tables and views API model.
  - Takeaway: table and view reads are expressed as HTTP resource requests with filters, selection, ordering, and pagination controls, not as frontend-authored raw SQL.
  - Applicability: this matches the app’s `.from(...).select(...).eq(...).range(...)` usage and supports the conclusion that the browser can log request intent reliably even when exact SQL is unavailable.
  - Adoption rationale: use the request-builder metadata as a first-class debugging artifact if the chosen scope stays frontend-only.
  - Rejection rationale: do not promise exact SQL text from browser instrumentation alone.

- Benchmark reference: PostgREST RPC model.
  - Takeaway: RPC functions are invoked through `/rpc/<function>` with named parameters and may execute any backend logic allowed by PostgreSQL.
  - Applicability: this app already uses multiple RPCs for date and cross-filter behavior, so the logging page must treat RPC calls as explicit loggable events rather than hiding them behind generic “query” rows.
  - Adoption rationale: log RPC name and parameters distinctly from table reads.
  - Rejection rationale: avoid flattening RPC calls into pseudo-table query rows because that would obscure their meaning during debugging.

- Benchmark reference: Supabase `pgaudit` guidance.
  - Takeaway: Supabase supports database-side audit logging that can expose backend statements, but it is role-scoped, operationally heavier, and not naturally bound to one browser session.
  - Applicability: this is the main viable path if the request truly requires backend SQL statements rather than client-visible query intent.
  - Adoption rationale: keep it as a fallback or alternative when exact SQL text is mandatory.
  - Rejection rationale: do not default to it for this request because it expands scope and weakens the “current session page” fit.

- Benchmark reference: Supabase `pg_stat_statements` guidance.
  - Takeaway: statement statistics can expose representative normalized queries and performance metadata, but they are aggregated database activity rather than a direct per-browser-session trace.
  - Applicability: useful for performance investigation, less suitable for showing exactly what the active app session just did.
  - Adoption rationale: useful as contextual evidence if performance debugging becomes relevant.
  - Rejection rationale: not the best default source for a session page aimed at debugging one user’s current app interaction.

## Minimal Spikes and Experiments
- **Spike: Local query-surface inventory**
  - Hypothesis: the existing React app has a narrow enough shared data-access surface that session logging can be added without editing every component individually.
  - Approach: inspect `App.jsx`, `supabase.js`, and `dataService.js`; search for Supabase query-builder and RPC usage across `react-app/src`.
  - Outcome: the dominant read paths are centralized in `fetchDimensions()` and exported data-service helpers, with App startup and report pages consuming those helpers.
  - Conclusion: shared instrumentation is feasible and is the lowest-risk implementation direction.

- **Spike: SQL visibility feasibility check**
  - Hypothesis: the current browser client path does not naturally expose literal backend SQL text for all existing queries.
  - Approach: compare the local code’s Supabase usage with PostgREST and Supabase observability documentation.
  - Outcome: table and RPC access are represented as PostgREST resource requests and function calls; true backend statement capture is documented as a database observability concern rather than a frontend feature.
  - Conclusion: the request needs one explicit design decision before implementation if exact SQL text is required.

## Multi-Perspective Stakeholder Review
### Senior Solution Architect
The request is technically feasible with a focused frontend change, but only if the implementation clearly distinguishes between client-visible request intent and exact backend SQL. The existing shared data layer is a strong anchor for a maintainable design, and the main architectural risk is choosing the wrong observability level.

- A shared instrumentation boundary is available and should be preferred over per-component logging.
- The unresolved semantics of “SQL queries” is the only material architecture fork.
- A DB-side auditing solution would be valid but is larger than the current request suggests.

### Product Owner
The business value is high because the page directly supports investigation of suspicious visual output without forcing developers to inspect code or network tools every time. The most important product quality attribute is clarity: the page must help explain what the app requested.

- The feature has diagnostic value even if it is not shown to every end user all the time.
- Scope is acceptable if it stays centered on the app session rather than a broader audit platform.
- Acceptance criteria should explicitly avoid misleading labels about exact SQL if only request intent is shown.

### User
From the user perspective, the new page is only useful if it turns confusing app behavior into understandable evidence. A dense dump of library internals would not satisfy the need to investigate incorrect visualized data.

- The page should explain query events in product terms such as target table or RPC and originating screen.
- Users benefit from seeing success and failure state, not just request metadata.
- Session continuity matters: the log should still reflect what happened after navigating across reports.

### Security Officer
This request is relatively safe when implemented as frontend session logging, but it still needs guardrails around what gets displayed and stored. Any shift toward database-side audit tooling would increase operational sensitivity.

- Do not expose credentials, auth tokens, or unnecessary payload values in the page.
- Keep session logging ephemeral unless persistence is explicitly required.
- If DB-side auditing is chosen, role-level logging and log retention must be treated as separate security decisions.

### Data Governance Officer
The request is about observability of data access, which can improve lineage understanding if done accurately. Governance risk appears when a log suggests it is authoritative SQL lineage while actually showing only a frontend approximation.

- The page should describe what level of fidelity it provides.
- RPC calls should be labeled distinctly because they can hide complex backend behavior behind a simple function name.
- Documentation should state any gap between client-visible requests and backend-executed SQL statements.
