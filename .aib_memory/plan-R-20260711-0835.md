# Implementation Plan

## Goal

Replace the complete Supabase-based architecture with Microsoft Azure-based architecture centered on Azure SQL Database while preserving all existing business functionality and user-visible behavior. The migration eliminates all production dependencies on Supabase, PostgreSQL, PostgREST, and the Supabase JavaScript client.

This migration is driven by strategic cloud infrastructure consolidation on Microsoft Azure. The target architecture implements a three-tier pattern: local Python ETL producing CSV outputs, Azure SQL Database serving as the cloud data warehouse, and Azure Functions providing a REST API layer between the database and the React frontend.

Impacted components include src/load_supabase.py (replaced with src/load_azure_sql.py), 11 PostgreSQL RPC functions (migrated to T-SQL stored procedures), React application data layer (react-app/src/lib/), all Supabase dependencies, menu.py integration, test suites, and project documentation. New components created include Azure infrastructure as code (Bicep templates), Azure Functions REST API, OpenAPI specification, T-SQL schema and stored procedures, and deployment automation.

## Constraints

- Existing Azure SQL infrastructure must be used: server sql-server-kolko-ni-struva, database sql-db-kolko-ni-struva
- SQL authentication required for ETL database connectivity (not Windows authentication)
- Managed Identity authentication required from Azure Functions to Azure SQL
- Anonymous read-only REST API without Azure API Management
- OpenAPI-first API design approach
- Avoid SQL Server compatibility level 170-specific features (compatibility level must be determined from existing database before SQL conversion)
- Configurable retention window via config.ini file with three-day default
- Hybrid REST endpoint design: GET /api/landing-page/rows, GET /api/landing-page/grouped, GET /api/options/{dimension}
- Structured error responses with error codes, messages, and correlation IDs
- Hybrid testing approach: data comparison tests plus critical workflow smoke tests
- Big-bang cutover deployment strategy
- ETL workflow stages extract.py and transform.py remain unchanged
- Local star schema structure and file formats remain unchanged
- React component logic and state management remain unchanged
- Netlify deployment process remains unchanged
- No changes to quality reporting logic or nomenclature files
- Autonomous decision: Target SQL Server compatibility level determined by querying existing database compatibility_level column before implementation begins

## Success criteria

- All Supabase dependencies removed from package.json, imports, and codebase
- All PostgreSQL-specific SQL constructs converted to T-SQL equivalents
- Azure SQL Database contains correct schema, indexes, and stored procedures
- load_azure_sql.py successfully synchronizes local CSV files to Azure SQL
- Azure Functions REST API operational and accessible from React application
- React application loads landing page data through Azure Functions endpoints
- Data comparison tests pass: all stored procedure results match corresponding Supabase RPC calls within tolerance
- Row counts match between local CSV files and Azure SQL tables
- Referential integrity validated between dimensions and facts
- Retention window correctly enforces three-day default from config.ini
- Critical user workflows function: landing page load, filtering, grouping, pagination
- Structured error responses displayed in React UI with user-friendly messages and correlation IDs
- menu.py Azure SQL sync option invokes load_azure_sql.py successfully
- Deployment scripts execute without errors and provision all Azure resources
- Rollback procedure documented and tested
- Documentation updated to reflect Azure architecture
- Architecture Decision Record documents Supabase-to-Azure capability mapping
